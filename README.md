> 💡 **1장은 앞으로 배우는 모든 네트워크의 "기초 뼈대"가 되는 중요한 단원입니다.**

## 🌐 1.1 네트워크 구성도 살펴보기

### 🏠 1.1.1 홈 네트워크
* **구성 흐름 :** `인터넷` ➡️ `공유기` ➡️ `PC` 또는 `스마트폰`
> 💡 **공유기의 역할:** 인터넷(외부 네트워크)과 집 안의 기기들(내부 네트워크)을 서로 연결하고 데이터를 중간에서 중개해 주는 핵심 장비입니다.

### 🏢 1.1.2 데이터 센터 네트워크 (3티어 아키텍처)
* **구성 흐름 :** `인터넷` ➡️ `방화벽` ➡️ `로드밸런서` ➡️ `WEB 서버` ➡️ `WAS 서버` ➡️ `DB 서버`
* **3티어 아키텍처(3-Tier Architecture)란?**
  * 웹 서버(WEB), 애플리케이션 서버(WAS), 데이터베이스(DB)를 각각 물리적·논리적으로 독립된 계층으로 쪼갠 구조입니다.
  * **장점 :** 서버가 철저히 분리되어 있어 외부 공격으로부터 **보안성**이 높고, 트래픽 분산(**성능**) 및 특정 계층만 수정하는 등의 관리(**유지보수**) 측면에서 매우 유리합니다.

## 🌐 1.2 프로토콜 : 네트워크 통신의 약속
* **프로토콜(Protocol)이란?** 컴퓨터가 네트워크망에서 특정 컴퓨터와 데이터를 주고받을 때 상호 간에 정의한 **일종의 약속이자 통신 양식**입니다.
* 프로토콜에는 이더넷, ICMP, IP, TCP, UDP 등 여러 종류가 있으며, 각 계층마다 사용하는 주소와 목적이 다릅니다.

### 📌 주요 프로토콜 핵심 요약
* **IP** : 목적지 장치를 찾아가기 위한 **주소 지정** 담당
* **TCP** : 패킷 분실 시 재전송을 통한 **신뢰성 보장**
* **UDP** : 확인 응답 없이 데이터를 빠르게 던지는 **속도 중심**
* **HTTP** : 텍스트 기반의 **웹 통신 표준**
* **HTTPS** : 보안 전송 플러그인을 더한 **암호화 웹 통신**
* **DNS** : 문자 도메인 이름을 숫자로 된 **IP 주소로 변환**
* **ARP** : 알고 있는 IP 주소를 기반으로 기기의 물리적인 **MAC 주소 조회**
* **ICMP** : `ping`과 같이 네트워크의 연결 상태 및 **오류 통보** 담당
* **SSH** : 암호화 통신을 이용한 안전한 **원격 제어 접속**

> 🏃‍♂️ **웹 접속 흐름 예시 :** `google.com` 입력 ➡️ **DNS**가 구글 서버 IP를 찾음 ➡️ **TCP** 3-way handshake 연결 확립 ➡️ **HTTP** 요청(Request) 전송 ➡️ 구글 서버가 화면 데이터를 담아 응답(Response)

## 층층이 쌓인 지도: 1.3 OSI 7계층과 TCP/IP
> 💡 **계층(Layer)으로 나누는 이유:** 통신 단계를 독립된 역할로 분리하면, 특정 네트워크 구간에 문제가 생겼을 때 **원인 분석과 유지보수가 훨씬 쉬워지기 때문**입니다.

<img width="929" height="765" alt="image" src="https://github.com/user-attachments/assets/883f5608-d7ba-4981-ac4a-4c1d4abd14b9" />

### 1.3.1 OSI 7계층 vs TCP/IP 모델 비교
실제 인터넷 환경에서는 OSI 7계층을 다소 단순화한 **TCP/IP 프로토콜 스택** 모델을 표준으로 사용합니다.

| OSI 7계층 (이론적 표준) | TCP/IP 4계층 (실무적 표준) | 주요 프로토콜 및 관련 장비 |
| :--- | :--- | :--- |
| **7계층: 응용 (Application)**<br>**6계층: 표현 (Presentation)**<br>**5계층: 세션 (Session)** | **4계층: 애플리케이션 (Application)** | HTTP, HTTPS, FTP, SSH, DNS |
| **4계층: 전송 (Transport)** | **3계층: 전송 (Transport)** | TCP, UDP (PDU 단위: *Segment*) |
| **3계층: 네트워크 (Network)** | **2계층: 인터넷 (Internet)** | IP, ICMP, 라우터, L3 스위치 (PDU: *Packet*) |
| **2계층: 데이터 링크 (Data Link)**<br>**1계층: 물리 (Physical)** | **1계층: 네트워크 인터페이스** | 이더넷, MAC 주소, L2 스위치, 랜선, 허브 (PDU: *Frame/Bit*) |

---

## 🔍 1.4 OSI 7계층별 상세 이해

### 🔌 1.4.1 1계층 - 물리 계층 (Physical Layer)
* **주요 역할 :** 상위 계층 데이터를 선로에 흘려보낼 수 있도록 **전기 신호로 변환하여 전달**합니다.
* **관련 장비 :** 랜 케이블, 허브(Hub), 리피터
* **데이터 단위 :** 비트 (Bit, 0과 1)

### 🎛️ 1.4.2 2계층 - 데이터 링크 계층 (Data Link Layer)
* **주요 역할 :** 물리적으로 연결된 **같은 네트워크(LAN) 내부에서 장비 간 통신**을 책임집니다.
* **주요 주소 :** MAC 주소 (하드웨어 고유 식별 주소)
* **관련 장비 :** L2 스위치 (Switch)
* **데이터 단위 :** 프레임 (Frame)
> 💡 **MAC 주소가 필요한 이유:** 같은 네트워크 대역 안에서 수많은 장비 중 정확히 누구에게 데이터를 전달할지 물리적으로 식별하기 위해서입니다.

### 🗺️ 1.4.3 3계층 - 네트워크 계층 (Network Layer)
* **주요 역할 :** 수많은 네트워크망이 얽힌 인터넷 환경에서 **목적지 대역을 찾아가는 경로(라우팅)를 결정**합니다.
* **주요 주소 :** IP 주소 (논리적 주소)
* **관련 장비 :** 라우터(Router), L3 스위치
* **데이터 단위 :** 패킷 (Packet)
> 💡 **IP 주소가 필요한 이유:** 전 세계 규모의 거대한 인터넷망에서 서로 다른 네트워크에 속한 목적지 컴퓨터의 대략적인 위치를 찾아가기 위함입니다.

### 📦 1.4.4 4계층 - 전송 계층 (Transport Layer)
* **주요 역할 :** 송신자와 수신자 간의 종단망 연결을 수립하고 **데이터 전송의 신뢰성을 제어**합니다.
* **주요 주소 :** 포트 번호 (Port Number)
* **프로토콜 종류 :** TCP(연결형, 신뢰성 보장), UDP(비연결형, 빠른 속도)
* **데이터 단위 :** 세그먼트 (Segment)

### 🔐 1.4.5 5계층 - 세션 계층 (Session Layer)
* **주요 역할 :** 양 끝단의 응용 프로세스가 통신을 하기 위한 **연결 세션을 생성, 유지, 종료(동기화)**하는 역할을 합니다.
* **예시 :** 웹 브라우저 로그인 상태 유지 메커니즘

### 🔏 1.4.6 6계층 - 표현 계층 (Presentation Layer)
* **주요 역할 :** 응용 계층이 데이터를 이해할 수 있도록 공통 형식을 정의하며, **데이터의 암호화, 압축, 인코딩**을 수행합니다.
* **예시 :** HTTPS 통신을 위한 SSL/TLS 암호화 처리, JPEG/MPEG 압축

### 💻 1.4.7 7계층 - 응용 계층 (Application Layer)
* **주요 역할 :** 사용자가 직접 사용하는 **소프트웨어 프로그램과 네트워크 서비스를 연결**해 주는 최상위 인터페이스입니다.
* **예시 :** 웹 브라우저(크롬), 이메일 프로그램, 메신저 등
* **대표 프로토콜 :** HTTP, HTTPS, FTP, SMTP, SSH

## 🎁 1.5 인캡슐레이션과 디캡슐레이션
데이터를 전송할 때는 위에서 아래로 내려가며 포장하고, 데이터를 받을 때는 아래에서 위로 올라가며 포장을 뜯습니다.

<img width="1340" height="701" alt="image" src="https://github.com/user-attachments/assets/0bcc2217-eb57-4fc8-91ef-2428eb4939b1" />

### 📦 1.5.1 인캡슐레이션 (Encapsulation)
* **정의 :** 송신 측 컴퓨터에서 데이터를 보낼 때, 상위 계층에서 하위 계층으로 내려가며 **각 계층의 통신 정보를 담은 '헤더(Header)'를 붙여 포장하는 과정**입니다.
* **흐름 :** `응용 데이터` ➡️ `TCP 헤더(포트 포함)` ➡️ `IP 헤더(IP 포함)` ➡️ `이더넷 헤더(MAC 포함)` ➡️ `전기 신호 변환 후 송신`

### 🔓 1.5.2 디캡슐레이션 (Decapsulation)
* **정의 :** 수신 측 컴퓨터가 데이터를 받았을 때, 물리 계층부터 시작해 상위 계층으로 올라가며 **자신에게 해당하는 '헤더'를 하나씩 검증하고 제거(포장을 해체)하는 과정**입니다.
* **흐름 :** `이더넷 헤더 제거(2계층 검증)` ➡️ `IP 헤더 제거(3계층 검증)` ➡️ `TCP 헤더 제거(4계층 검증)` ➡️ `최종 응용 데이터 추출 및 프로그램 전달`

## 🛣️ 2.2 네트워크 회선 종류
> **회선이란?** 데이터가 이동하는 네트워크의 물리적·논리적 연결 통로입니다.

### 1️⃣ 인터넷 회선 (공용 회선)
* **특징 :** 하나의 회선을 **여러 사용자가 공유**하여 사용하는 일반적인 인터넷망입니다.
* **장단점 :** 비용이 매우 저렴하지만, 사용자가 몰리는 시간대에는 속도 저하나 품질 저하가 발생할 수 있습니다.

### 2️⃣ 전용 회선 (Leased Line)
* **특징 :** 가입자와 통신사 간을 가상 혹은 물리적으로 1:1 전용 연결하여 **특정 기업만 독점 사용하는 통신선**입니다.
* **장단점 :** 대역폭이 완전히 보장되어 안정성이 최고조이며 보안성이 완벽하지만, 비용이 매우 비쌉니다.

### 3️⃣ 인터넷 전용 회선
* **특징 :** 인터넷 접속용 회선이지만, 일반 가정용과 달리 **기업이 회선 대역폭을 전용(독점)처럼 보장받아 사용하는 방식**입니다.
* **장단점 :** 일반 인터넷보다 품질이 훨씬 안정적이며, 전용 회선보다는 저렴하여 기업용 메인 회선으로 가장 많이 쓰입니다.

### 4️⃣ VPN (Virtual Private Network, 가상 사설망)
* **특징 :** 퍼블릭 인터넷 회선 위에 **터널링과 암호화 기술을 적용해 "가상 전용망"처럼 안전하게 만드는 기술**입니다.
* **용도 :** 인터넷은 본래 도청과 해킹 위험에 노출되어 있습니다. VPN을 통해 패킷을 암호화함으로써 **재택근무자나 원격 지사가 본사 내부망에 안전하게 접속**할 수 있도록 합니다.

### 5️⃣ DWDM (Dense Wavelength Division Multiplexing)
* **특징 :** 단 한 가닥의 광케이블 안에 **여러 개의 서로 다른 빛의 파장(색상)을 실어 데이터를 동시에 전송**하는 고밀도 파장 분할 다중화 기술입니다.
* **용도 :** 초고속·대용량 데이터 전송이 필수적인 **대형 통신사, 글로벌 데이터 센터(IDC), 국가 간 해저 케이블** 등에 필수적으로 사용됩니다.

## 🛠️ 2.3 네트워크 핵심 구성 요소

### 💳 2.3.1 NIC (Network Interface Card, 랜카드)
* **정의 :** 컴퓨터나 서버를 네트워크선에 물리적으로 연결해 주는 확장 카드 장치입니다.
* **역할 :** 기기 고유의 물리 주소인 **MAC 주소를 보유**하고 있으며, 컴퓨터 내부의 디지털 데이터를 네트워크용 전기/광 신호로 상호 변환하여 송수신합니다.

### 🧷 2.3.2 케이블과 커넥터
<img width="590" height="379" alt="image" src="https://github.com/user-attachments/assets/3a3fbc1b-a715-4b3a-b5b2-0312b26c2617" />

* **UTP 케이블 :** 구리선 8가닥을 꼬아서 만든 일반적인 LAN 케이블입니다. 저렴하고 다루기 쉽지만 전송 거리가 짧습니다(약 100m 제한).
* **광케이블 (Fiber Optic) :** 빛 신호를 이용하여 데이터를 전송하는 케이블입니다. 속도가 매주 빠르고 감쇄가 없어 수십 km 이상의 **장거리 대용량 전송**이 가능합니다.
* **RJ-45 커넥터 :** UTP 케이블 끝에 찍어서 컴퓨터 랜 포트에 딱 소리가 나게 꽂는 표준 플라스틱 커넥터입니다.

### 🔌 2.3.3 허브 (Hub)
* **특징 :** 전기 신호를 단순히 증폭하여 연결된 **모든 포트로 무조건 데이터를 복제 전송**하는 1계층 장비입니다.
* **문제점 :** 원하는 목적지가 아닌 포트에도 트래픽이 전부 전달되므로 데이터 충돌(Collision)이 잦고 보안성이 떨어져 **현재 실무에서는 거의 사용하지 않습니다.**

### 🎛️ 2.3.4 스위치 (Switch)
<img width="1000" height="1000" alt="image" src="https://github.com/user-attachments/assets/58cf9348-0157-49e3-90a2-9112e96e592b" />

* **특징 :** 허브와 달리, 각 목적지의 **MAC 주소를 스스로 학습하여 필요한 타깃 포트로만 데이터를 전송**하는 2계층 장비입니다.
* **장점 :** 불필요한 트래픽 유출을 막아 데이터 충돌이 획기적으로 감소하며, 통신 효율성과 속도가 매우 뛰어납니다.

### 🗺️ 2.3.5 라우터 (Router)
<img width="600" height="338" alt="image" src="https://github.com/user-attachments/assets/4b331414-3762-4a08-98cd-560fd45835cb" />

* **특징 :** LAN의 경계선에 위치하여 **서로 다른 네트워크 대역을 상호 연결**해 주는 3계층 장비입니다.
* **역할 :** 들어온 패킷의 IP 주소를 확인하고, 내부 '라우팅 테이블'을 참조하여 목적지까지 가는 가장 빠르고 정확한 경로를 지정(Routing)해 줍니다.

### ⚖️ 2.3.6 로드 밸런서 (Load Balancer)
<img width="800" height="320" alt="image" src="https://github.com/user-attachments/assets/7186da9b-c5c2-4f22-b956-6bbb6d5578cc" />

* **특징 :** 외부 사용자의 대규모 접속 요청(트래픽)을 한 곳으로 몰리지 않게 **여러 대의 서버로 골고루 나누어 배분하는 부하분산 장비**입니다.
* **필요성 :** 서비스 규모가 커지면 서버 1대만으로는 감당할 수 없습니다. 로드 밸런서 뒤에 동일한 서버 여러 대를 배치하여 시스템이 죽지 않고 안정적으로 버틸 수 있게 만듭니다.

### 🛡️ 2.3.7 보안 장비 (방화벽 및 IPS)
* **방화벽 (Firewall) :** 사전에 정의된 보안 규칙을 기반으로, **특정 IP 주소나 포트 번호를 식별하여 트래픽을 허용(Allow)하거나 차단(Deny)**하는 경계형 보안 장비입니다.
* **IPS (Intrusion Prevention System, 침입 방지 시스템) :** 방화벽을 통과해 유입되는 패킷의 패턴을 실시간 깊숙이 분석하여, 악성코드나 **해킹 시도(취약점 공격 등)의 유해성을 직접 감지하고 즉각 차단**하는 능동형 보안 장비입니다.

### 📦 2.3.8 기타 장비
* **모뎀 (Modem) :** 아날로그 신호와 디지털 신호를 상호 변환하여, 외부 통신사망 인프라의 신호를 컴퓨터가 쓸 수 있게 바꾸어 주는 장치입니다.
* **공유기 (IP Router) :** 가정이나 소규모 사무실용 올인원 장비입니다. 외부 공인 IP 하나를 나누어 쓰는 주소 변환(**NAT**), 내부 IP 자동 대여(**DHCP**), 무선 인터넷(**Wi-Fi**), 그리고 기초적인 라우팅 기능까지 하나의 기기에 압축해 놓은 소형 네트워크 장비입니다.

# 3장 네트워크 통신하기
> 💡 **3장에서는 컴퓨터들이 서로를 어떻게 식별하고, 실제로 데이터를 주고받는 방식과 프로토콜의 원리를 상세히 알아봅니다.**

---

## 📢 3.1 데이터 전송 방식 (유니·브로드·멀티·애니캐스트)
> 네트워크 안에서 데이터를 **"누구에게, 어떤 범위로 보내는가"**에 따른 분류입니다.

### 👥 3.1.1 유니캐스트 (Unicast)
* **정의 :** 고유한 하나의 목적지만을 지정하여 통신하는 **1:1 전송 방식**입니다.
* **특징 :** 수신지 MAC/IP 주소가 명확히 지정되어 있어 다른 장비들에게 트래픽 부담을 주지 않습니다.
* **예시 :** 카카오톡 1:1 메시지 전송, 일반적인 웹 서핑(HTTP) 등

### 📢 3.1.2 브로드캐스트 (Broadcast)
* **정의 :** 자신이 속한 **동일 네트워크 대역 전체에 데이터를 뿌리는 1:All 전송 방식**입니다.
* **특징 :** 목적지 주소가 통상 `FF:FF:FF:FF:FF:FF` 등으로 고정되며, 스위치는 이 패킷을 모든 포트로 복제(Flooding)합니다. 트래픽이 과도해지면 네트워크 성능 저하를 유발합니다.
* **예시 :** 상대방 물리 주소를 찾기 위한 ARP 요청, IP 자동 할당을 위한 DHCP 요청

### 🍿 3.1.3 멀티캐스트 (Multicast)
* **정의 :** 특정 멀티캐스트 그룹 주소에 가입한 **특정 장비들에게만 데이터를 전송하는 1:Group 전송 방식**입니다.
* **특징 :** 브로드캐스트처럼 전체에 부하를 주지 않고, 유니캐스트처럼 수백 명에게 개별로 보낼 필요가 없어 대역폭 효율성이 매우 높습니다.
* **예시 :** IPTV 실시간 방송 송출, 대규모 화상 회의 시스템

### 📍 3.1.4 애니캐스트 (Anycast)
* **정의 :** 동일한 IP 주소를 전 세계 여러 서버가 공유하고, 사용자와 **지리적 또는 네트워크 상으로 가장 가까운 '단 한 대'의 서버가 응답하는 1:1 전송 방식**입니다.
* **특징 :** 서버 장애 시 자동으로 다음으로 가까운 정상 서버가 매핑되므로 가용성과 응답 속도가 극대화됩니다.
* **예시 :** 글로벌 CDN(콘텐츠 전송 네트워크), 전 세계 루트 DNS 서버 구조

---

## 🪪 3.2 MAC 주소 (물리 주소)
* **정의 :** 네트워크 카드(NIC) 제조업체가 출하 시 하드웨어에 고정하여 부여하는 **전 세계 유일한 2계층 물리 주소**입니다.
* **표기 및 크기 :** 총 **48비트(6바이트)** 크기이며, `00:1A:2B:3C:4D:5E`와 같이 16진수 12자리로 표기합니다.
* **구조 분해 :**
  * **앞 24비트 (OUI) :** 하드웨어 제조사(예: Intel, Realtek 등)를 식별하는 고유 코드
  * **뒤 24비트 (UAA) :** 제조사에서 각 장비에 부여한 고유 시리얼 번호
* **동작 원리 :** 인터넷 전체 주소와 상관없이, **동일한 LAN(같은 네트워크 대역) 내부에서 패킷이 실제로 움직일 때는 오직 이 MAC 주소만을 기반으로 통신**이 이루어집니다.

---

## 🗺️ 3.3 IP 주소 (논리 주소)
* **정의 :** 단말이 연결된 네트워크의 위치에 따라 유연하게 변경하고 할당할 수 있는 **3계층 논리 주소**입니다. 인터넷 세상의 최종 출발지와 목적지를 식별합니다.

### 🔢 3.3.1 IP 주소 체계 비교
* **IPv4 :** 32비트 체계이며 점(`.`)으로 구분된 4개의 옥텟(예: `192.168.0.1`)으로 표기합니다. (약 43억 개 주소 제한으로 현재 고갈 상태)
* **IPv6 :** IPv4 고갈을 해결하기 위한 128비트 체계이며, 콜론(`:`)으로 구분된 16진수 형태로 표기합니다. (거의 무한대에 가까운 주소 제공)

### 📊 3.3.2 클래스풀(Classful)과 클래스리스(Classless)
과거 방식인 클래스풀의 낭비를 막기 위해 현대에는 모두 클래스리스 방식을 사용합니다.

* **클래스풀 (Classful) :** 주소의 맨 앞자리 숫자를 보고 네트워크 크기를 고정 패턴(A, B, C 클래스)으로 딱 잘라 분류하던 구식 방식입니다.
  * **A 클래스 :** 대형망 (`0.0.0.0` ~ `127.255.255.255` / 서브넷 마스크: `255.0.0.0`)
  * **B 클래스 :** 중형망 (`128.0.0.0` ~ `191.255.255.255` / 서브넷 마스크: `255.255.0.0`)
  * **C 클래스 :** 소형망 (`192.0.0.0` ~ `223.255.255.255` / 서브넷 마스크: `255.255.255.0`)
  * **단점 :** IP가 100개만 필요한 소기업에 C클래스(256개)를 주면 156개가 낭비되고, 300개가 필요한 중소기업에 B클래스(65,536개)를 주면 수만 개의 IP 주소가 통째로 버려지는 치명적인 비효율성이 있었습니다.
* **클래스리스 (Classless) :** 클래스의 고정 개념을 버리고, **CIDR(Classless Inter-Domain Routing) 표기법을 도입하여 서브넷 마스크 값 표기(예: `/24`)를 통해 필요한 크기만큼 자유자재로 네트워크를 쪼개어 할당**하는 현대 표준 방식입니다.

### 📐 3.3.3 서브네팅 (Subnetting)의 원리와 가치
* **개념 :** 하나의 큰 네트워크 대역을 서브넷 마스크를 조절하여 **여러 개의 작고 독립된 서브 네트워크(Subnet)로 쪼개는 기술**입니다. (예: 사내 망을 개발팀, 인사팀, 서버망으로 각각 분리)
* **장점 :** 불필요한 브로드캐스트 트래픽 전송 범위 차단(**성능 향상**), 팀 간 접근 제어 적용(**보안 강화**), 효율적인 IP 주소 분배.
* **⚠️ 실무 계산 핵심 규칙 (IP 낭비 방지) :**
  * 어떤 서브넷 대역이든 **맨 첫 번째 주소(네트워크 ID)**와 **맨 마지막 주소(브로드캐스트 주소)** 2개는 고유 장비(PC/서버)에 직접 할당할 수 없습니다.
  * *예시:* `192.168.0.0/24` 대역에서 `192.168.0.0`은 이 대역 전체를 대표하는 이름(네트워크 고유 ID)이며, `192.168.0.255`는 이 대역 전체에 외치는 브로드캐스트 주소입니다. 따라서 실제 호스트가 쓸 수 있는 주소는 `192.168.0.1` ~ `192.168.0.254`까지 총 254개입니다.

### 🔐 3.3.4 공인 IP와 사설 IP
* **공인 IP (Public IP) :** 전 세계에서 유일해야 하며 통신 사업자(ISP)가 할당합니다. **공용 인터넷망에서 서로 직접 라우팅이 가능**한 실제 외부 주소입니다.
* **사설 IP (Private IP) :** 회사 사내망이나 가정용 공유기 내부 등 **독립된 사설 네트워크 안에서만 통신용으로 자유롭게 나눠 쓰는 주소**입니다.
  * 대표 대역: `10.x.x.x`, `172.16.x.x` ~ `172.31.x.x`, `192.168.x.x`
  * 사설 IP 기기는 외부 인터넷망에 직접 패킷을 보낼 수 없기 때문에, 반드시 **NAT(주소 변환) 장비**를 거쳐 공인 IP로 옷을 갈아입고 나가야 합니다.

## 📦 3.4 전송 제어 프로토콜 (TCP와 UDP)

### 🤝 3.4.1 TCP (Transmission Control Protocol)
* **특징 :** 안정성과 정확성이 최우선인 **연결 지향형 프로토콜**입니다. 패킷의 유실 검사, 흐름 제어, 데이터 전송 순서 보장, 에러 재전송 메커니즘이 꼼꼼하게 들어가 있어 안전한 반면 헤더가 무겁고 상대적으로 느립니다.
* **용도 :** 웹 브라우징(HTTP/HTTPS), 파일 전송(FTP), 원격 터미널(SSH), 이메일 등 데이터 유실이 단 1비트도 용납되지 않는 서비스.
* **🤝 TCP 3-Way Handshake (연결 확립 절차) :**
  1. **클라이언트 $\rightarrow$ 서버 (SYN) :** "똑똑, 제 포트 정보 보낼 테니 연결 가능합니까?"
  2. **서버 $\rightarrow$ 클라이언트 (SYN + ACK) :** "네, 요청 확인했습니다. 저도 포트 준비되었으니 연결 수락합니다."
  3. **클라이언트 $\rightarrow$ 서버 (ACK) :** "확인했습니다! 지금부터 신뢰 통로 개설하고 진짜 데이터 보냅니다."

### 🚀 3.4.2 UDP (User Datagram Protocol)
* **특징 :** 신뢰성을 위한 사전 연결이나 확인 응답을 완벽히 배제한 **비연결형 프로토콜**입니다. 일방적으로 데이터를 연속해서 던지기 때문에 오버헤드가 극도로 적고 전송 속도가 압도적으로 빠릅니다. 단, 중간에 데이터가 유실되어도 복구하지 않습니다.
* **용도 :** 온라인 실시간 게임 패킷, 실시간 스트리밍 방송, 음성 통화(VoIP), 가벼운 도메인 조회(DNS) 등 속도와 연속성이 유실 대응보다 더 중요한 서비스.

### 🚪 3.4.3 포트 번호 (Port Number)의 개념
* **역할 :** 하나의 IP 주소(컴퓨터) 안에서 동시에 작동하고 있는 **여러 개의 네트워크 애플리케이션 서비스들을 논리적으로 완벽히 구분**하기 위한 식별 방 번호입니다.
> 💡 **쉽게 이해하기:** IP 주소가 아파트의 전체 '지번 주소'라면, 포트 번호는 그 아파트의 구체적인 '동·호수'에 해당하여 최종적으로 어떤 앱(웹 브라우저인지, 메신저인지)에 데이터를 배달해야 할지 결정합니다.

* **대표적인 표준 Well-Known 포트 번호 :**
  * `21` : FTP (파일 전송)
  * `22` : SSH (보안 원격 제어 접속)
  * `53` : DNS (도메인 주소 해석)
  * `80` : HTTP (일반 웹 서비스)
  * `443` : HTTPS (보안 암호화 웹 서비스)

---

## 🔍 3.5 ARP (주소 변환 프로토콜)

### 3.5.1 ARP (Address Resolution Protocol)의 존재 이유
* 목적지 단말의 **논리 주소(IP)는 알고 있으나, 로컬 통신을 위한 하드웨어 물리 주소(MAC)를 모를 때** 이를 자동으로 알아내어 연결해 주는 핵심 프로토콜입니다.

### 3.5.2 ARP 기본 동작 4단계
1. **ARP 요청 (Request) :** 송신 장비가 동일 대역 전체에 "IP `192.168.0.1` 쓰시는 장비님, MAC 주소 좀 알려주세요!"라고 **브로드캐스트**를 날립니다.
2. **ARP 응답 (Reply) :** 해당 IP를 실제로 사용하는 단말(예: 게이트웨이 라우터)만 이를 감지하고, 자신의 MAC 주소를 담아 요청자에게 1:1 **유니캐스트**로 답변을 돌려줍니다.
3. **ARP 캐시 테이블 저장 :** 송신 장비는 받아온 IP-MAC 매핑 정보를 운영체제 내부 메모리(ARP Table)에 기록해 둡니다. (`arp -a` 명령어로 확인 가능)
4. **실제 데이터 통신 :** 이후부터는 브로드캐스트 없이 저장된 테이블 항목을 보고 직접 목적지 MAC 주소 헤더를 붙여 초고속 L2 통신을 진행합니다.

### 📢 3.5.3 GARP (Gratuitous ARP)
* **정의 :** 상대방이 요청하지 않았는데도, **자기 자신의 IP와 MAC 주소 정보를 네트워크 전체에 자발적으로 알리는 특수 ARP 패킷**입니다.
* **용도 :** * 로컬 대역에 나랑 똑같은 IP를 몰래 쓰는 장비가 있는지 검사 (**IP 중복 확인**)
  * 이중화(HA) 구성된 방화벽/라우터 중 메인 장비가 다운되어 서브 장비로 마스터 권한이 넘어갈 때, 하단 스위치들의 MAC 주소 장부를 일제히 갱신시키기 위한 용도 (**Failover 알림**)

### 🔄 3.5.4 RARP (Reverse ARP)
* **정의 :** ARP와 완전히 정반대로, 자신의 **물리 주소(MAC)는 알지만 논리 주소(IP)를 모를 때 IP를 요청**하는 구식 프로토콜입니다.
* **현황 :** 하드웨어 디스크가 없는 더미 터미널의 초기 구동 시 쓰였으나, 현대에는 기능이 훨씬 고도화된 **BOOTP 및 DHCP 프로토콜로 완전히 대체**되어 실무에서는 거의 자취를 감추었습니다.

## 🚪 3.6 서브넷과 기본 게이트웨이 (Default Gateway)

### 3.6.1 게이트웨이의 본질과 통신 분기 흐름
* **정의 :** 자신이 속한 로컬 네트워크(LAN)의 울타리를 벗어나, **외부 인터넷망이나 완전히 다른 대역의 네트워크로 패킷을 내보낼 때 반드시 통과해야 하는 관문(출입구)**입니다. 보통 해당 대역 라우터의 내부 인터페이스 IP 주소가 지정됩니다.

* **💻 상황별 통신 흐름 분석 :**
  * **상황 A. 내부(같은 서브넷) 통신일 때 :**
    * *조건:* PC1(`192.168.0.10`) $\rightarrow$ PC2(`192.168.0.20/24`)로 전송
    * *판단:* 서브넷 마스크 확인 결과 두 기기는 같은 대역입니다. 게이트웨이를 거치지 않고 로컬 **L2 스위치를 통해 MAC 주소 기반으로 1:1 직접 다이렉트 통신**을 완료합니다.
  * **상황 B. 외부(다른 서브넷) 통신일 때 :**
    * *조건:* PC1(`192.168.0.10`) $\rightarrow$ 구글DNS 서버(`8.8.8.8`)로 전송
    * *판단:* 목적지가 완전히 다른 대역입니다. PC1은 패킷을 직접 보낼 방법이 없으므로, 무조건 자신의 설정에 박혀 있는 **기본 게이트웨이(라우터) IP 주소로 패킷을 던집니다.** 이후 라우터가 외부망 릴레이를 책임집니다.

### ⚖️ 3.6.2 2계층 통신 vs 3계층 통신 한눈에 비교
네트워크 엔지니어로서 실무 장애 처리를 할 때 통신 단계를 명확히 분리하는 기준점입니다.

| 비교 항목 | 2계층 통신 (L2 통신) | 3계층 통신 (L3 통신) |
| :--- | :--- | :--- |
| **통신 범위** | 동일 LAN 영역 내부 (동일 서브넷 대역) | 서로 다른 독립 네트워크 간 (원격지 인터넷) |
| **핵심 식별 주소**| 목적지 장비의 **MAC 주소** | 최종 타깃 장비의 **IP 주소** |
| **중계 장비** | L2 스위치 (Switch), 허브 | 라우터 (Router), L3 스위치 |
| **동작 특징** | MAC 주소 테이블 기반 포워딩, 매우 빠름 | 라우팅 테이블 경로 분석, 게이트웨이 필수 통과 |
  
## 4장 스위치 : 2계층 장비 정리

### 4.1 스위치 장비 동작
* 스위치는 **2계층(Data Link Layer) 장비**입니다.
* **MAC 주소**를 기반으로 동작합니다.
* 허브(Hub)는 수신한 프레임을 모든 포트로 복사해서 뿌리지만, 스위치는 **필요한 포트로만 타깃 전달**합니다.

#### 4.1.1 플러딩 (Flooding)
* 스위치가 목적지 MAC 주소를 모를 때 발생합니다.
* 수신한 포트를 제외한 **모든 포트로 프레임을 전송**하는 것을 플러딩이라고 합니다.
* 초기에는 MAC 주소 테이블이 비어있으므로 전체에 보내고, 이에 대한 응답을 받으면서 MAC 주소를 학습합니다.

#### 4.1.2 어드레스 러닝 (Address Learning)
* 스위치가 프레임의 **출발지(Source) MAC 주소**를 보고 해당 포트에 어떤 장비가 연결되어 있는지 학습하는 과정입니다.
* 학습한 정보는 스위치의 **MAC 주소 테이블(MAC Address Table)**에 저장됩니다.

#### 4.1.3 포워딩 (Forwarding) / 필터링 (Filtering)
* **포워딩 :** 목적지 MAC 주소의 위치를 알고 있다면, 해당 포트로만 프레임을 정확히 전송합니다.
* **필터링 :** 목적지가 아닌 다른 불필요한 포트로는 프레임을 전송하지 않아 네트워크 효율을 높입니다.

> **💡 스위치 핵심 동작 흐름**
> 1. 프레임 수신
> 2. 출발지(Source) MAC 주소 학습
> 3. 목적지(Destination) MAC 주소 확인
> 4. 테이블에 목적지가 **있으면 포워딩**, **없으면 플러딩**

---

### 4.2 VLAN (Virtual Local Area Network)

#### 4.2.1 VLAN이란?
* 하나의 물리적 스위치를 여러 개의 **논리적인 네트워크로 분리**하는 기술입니다.
* **사용 목적 :** 하나의 스위치 안에서 개발팀, 인사팀, 서버망 등을 서로 분리하여 관리할 수 있습니다.
* **VLAN의 장점 :**
    1.  **보안 향상 :** 다른 VLAN(부서) 간의 무분별한 통신을 기본적으로 제한합니다.
    2.  **브로드캐스트 감소 :** 브로드캐스트가 퍼지는 범위(Broadcast Domain)를 축소시켜 네트워크 성능을 향상합니다.
    3.  **관리 편리 :** 물리적인 장비나 케이블 추가 없이 소프트웨어 설정만으로 논리적 분리가 가능합니다.

> * **VLAN이 없을 때 :** 스위치에 연결된 모든 장비가 같은 브로드캐스트 도메인에 속하므로 성능 저하 및 보안 취약점이 발생합니다.
> * **VLAN이 있을 때 :** `VLAN 10(개발팀)`, `VLAN 20(인사팀)`처럼 논리적으로 완전히 분리되어 다른 네트워크처럼 동작합니다.

#### 4.2.2 VLAN의 종류와 특징
* **Static VLAN (포트 기반 VLAN) :** 스위치 포트에 직접 VLAN 번호를 지정하는 방식이며, 가장 일반적입니다. (ex. 1번 포트 = VLAN 10)
* **Dynamic VLAN (맥/IP 기반 VLAN) :** 접속하는 장비의 조건(MAC 주소 등)을 기반으로 자동으로 할당하는 방식입니다. 실무에서는 상대적으로 적게 사용됩니다.
* **VLAN ID :** VLAN을 구분하는 번호입니다. (ex. VLAN 10, VLAN 20, VLAN 100)

#### 4.2.3 VLAN 모드 (Access / Trunk) 동작 방식
* **Access Port (액세스 포트) :** * **하나의 VLAN**만 통과할 수 있는 포트입니다.
    * 보통 PC, 프린터 등 단말 장비를 연결할 때 사용합니다.
    * 프레임에 VLAN 태그가 붙지 않습니다. *(비유: 특정 부서 전용문)*
* **Trunk Port (트렁크 포트) :** * **여러 개의 VLAN** 프레임이 동시에 통과할 수 있는 포트입니다.
    * 보통 `스위치 <-> 스위치` 또는 `스위치 <-> 라우터/서버` 연결용으로 사용합니다.
    * 어느 VLAN의 프레임인지 구분하기 위해 프레임에 **VLAN 태그(IEEE 802.1Q 표준)**를 붙여서 전송합니다. *(비유: 여러 부서 공용 통로)*

---

### 4.3 STP (Spanning Tree Protocol)

#### 4.3.1 루프(Loop)란?
* 네트워크의 가용성(이중화)을 위해 스위치를 이중으로 연결(`SW1 <-> SW2` 간 회선 2개 이상)했을 때 발생할 수 있는 현상입니다.
* **루프 발생 시 문제점 :**
    1.  **브로드캐스트 폭주 (Broadcast Storm) :** 프레임이 네트워크 내부에서 무한 반복 돌면서 전체 네트워크가 마비됩니다.
    2.  **MAC 테이블 꼬임 (MAC Table Instability) :** 스위치가 동일한 MAC 주소의 위치를 양쪽 포트에서 번갈아 학습하여 혼동을 겪습니다.
    3.  **CPU 과부하 :** 무한 루프 도는 프레임 처리로 인해 장비 리소스가 고갈되어 다운될 수 있습니다.
* **주의 :** 3계층 패킷과 달리 **2계층 프레임에는 TTL(Time To Live) 개념이 없기 때문에**, 한 번 루프가 돌기 시작하면 물리적으로 연결을 끊거나 장비를 끄기 전까지 멈추지 않습니다.

#### 4.3.2 STP란?
* **Spanning Tree Protocol :** 루프를 방지하기 위해 만들어진 2계층 프로토콜입니다.
* **동작 방식 :** 스위치끼리 제어 메시지(BPDU)를 주고받으며 협의한 후, 루프를 유발하는 **특정 포트를 논리적으로 차단(Block)**하여 구조를 트리 형태로 만듭니다.
* **특징 :** 루프는 막으면서 물리적 이중화는 유지하므로, 사용 중인 회선에 장애가 발생하면 차단되었던 포트가 **자동으로 활성화**되어 백업 경로로 동작합니다.

> **⚙️ STP 동작 3단계 흐름**
> 1. **Root Bridge 선출 :** 네트워크의 기준이 될 대장 스위치를 선택합니다. (Bridge ID의 우선순위가 가장 높은 장비)
> 2. **최적 경로 계산 :** 각 스위치에서 Root Bridge까지 가는 가장 최단 경로(Cost가 가장 낮은 경로)를 계산합니다.
> 3. **일부 포트 차단 :** 최적 경로 외에 루프를 유발하는 나머지 포트를 차단합니다.

* **STP 포트 상태 변화 단계 :**
    `Blocking` (차단) ➡️ `Listening` (준비) ➡️ `Learning` (MAC 학습) ➡️ `Forwarding` (정상 송수신)

#### 4.3.3 향상된 STP 종류
* **RSTP (Rapid STP) :** * 기존 STP의 느린 수렴 속도(장애 복구 시 약 30~50초 소요)를 개선하여 **수초 이내로 빠르게 복구**하는 프로토콜입니다. 현재 실무에서 가장 널리 사용됩니다.
* **MSTP / MST (Multiple STP) :** * VLAN이 많을 때 모든 VLAN마다 STP를 돌리면 장비 부하가 심해지므로, **VLAN들을 몇 개의 그룹(Instance)으로 묶어서** 효율적으로 STP를 구성하는 방식입니다. 대규모 대기업/엔터프라이즈 환경에서 주로 사용됩니다.
  

## 5장 라우터/L3 스위치 : 3계층 장비
> **핵심 과제 :** 서로 다른 네트워크끼리 어떻게 연결하고 통신할 것인가?

### 5.1 라우터의 동작 방식과 역할
* 라우터는 **3계층(Network Layer) 장비**입니다.
* **IP 주소**를 기반으로 동작하며, **서로 다른 네트워크를 연결**하는 역할을 합니다.

#### 5.1.1 경로 지정 (Routing)
* 라우터의 가장 핵심적이고 중요한 기능입니다.
* 목적지 IP까지 패킷을 어디로 보내야 가장 빠르고 정확한지 판단하는 과정을 **경로 지정(Routing)**이라고 합니다.
* **동작 예시 :** `PC(192.168.0.10)`가 목적지 `8.8.8.8`로 패킷을 보낼 때, 서브넷 마스크를 비교하여 서로 다른 네트워크임을 감지하고 기본 게이트웨이(라우터)로 전달합니다. 라우터는 목적지 네트워크를 확인 후 최적의 경로로 패킷을 던집니다.
* **라우팅 테이블 :** 목적지 네트워크별 경로 정보를 저장해 둔 일종의 '길찾기 지도'입니다.

#### 5.1.2 브로드캐스트 컨트롤 (Broadcast Control)
* 라우터는 **브로드캐스트 패킷을 차단(Drop)**하여 다른 네트워크로 넘어가지 않게 합니다.
* 브로드캐스트가 전 세계 네트워크로 퍼지면 전체 폭주 및 성능 저하가 발생하기 때문입니다.
* **스위치와의 차이점 :** * **스위치 :** 같은 VLAN 내부 안에서는 브로드캐스트를 모두에게 전달(플러딩)합니다.
    * **라우터 :** 브로드캐스트 영역(Broadcast Domain)을 물리적으로 나누어 차단하므로 네트워크를 안정적이고 효율적으로 유지합니다.

#### 5.1.3 프로토콜 변환
* 서로 다른 네트워크 기술이나 규격을 연결할 수 있도록 변환해 줍니다.
* LAN 구간의 `Ethernet` 환경과 외부 `WAN` 환경을 연결하거나, `IPv4`와 `IPv6` 간의 통신이 가능하도록 일종의 **'통역사' 역할**을 수행합니다.

---

### 5.2 경로 지정 - 라우팅/스위칭

#### 5.2.1 라우팅 동작과 라우팅 테이블
* 라우터가 패킷을 수신했을 때 실행하는 내부 메커니즘입니다.
> **⚙️ 라우터의 패킷 처리 과정**
> 1. 패킷 수신 후 **목적지 IP 주소** 확인
> 2. 내부 **라우팅 테이블** 조회
> 3. 가장 적절한(구체적인) **최적 경로 선택**
> 4. 다음 장비(Next Hop)나 출력 인터페이스로 **패킷 전달**

* **라우팅 테이블 포함 요소 :** 목적지 네트워크, 서브넷 마스크, 다음 홉(Next Hop), 나가는 인터페이스(Output Interface), 메트릭(Metric) 등

#### 5.2.2 라우팅 (라우터가 경로 정보를 얻는 방법)
라우터가 목적지 정보를 학습하는 방법은 크게 3가지로 나뉩니다.

1. **다이렉트 커넥티드 (Direct Connected) :**
    * 라우터 자신의 인터페이스에 케이블로 직접 연결된 네트워크입니다. IP와 서브넷 마스크를 설정하면 **자동으로 학습**됩니다.
2. **스태틱 라우팅 (Static Routing) :**
    * 관리자가 특정 목적지 경로를 명령어로 직접 입력하는 방식입니다.
    * *설정 예시 :* `ip route 10.0.0.0 255.255.255.0 192.168.1.1` (`10.0.0.0/24` 네트워크로 가려면 `192.168.1.1`로 보내라)
3. **다이나믹 라우팅 (Dynamic Routing) :**
    * 실무 및 대규모 네트워크의 핵심입니다. 라우팅 프로토콜을 통해 라우터끼리 **자동으로 네트워크 정보를 교환**합니다.
    * 장애 발생 시 우회 경로를 자동으로 계산하며, 대표적인 프로토콜은 다음과 같습니다.
        * **OSPF :** 엔터프라이즈 실무에서 가장 중요한 링크 상태(Link-State) 기반 프로토콜로, 빠르고 효율적입니다.
        * **BGP :** 전 세계 인터넷을 연결하는 대규모 외장형 프로토콜로, 통신사(ISP) 간 라우팅에 사용됩니다.
        * *기타 :* RIP, EIGRP 등

#### 5.2.3 스위칭 (라우터가 패킷을 뱁어내는 과정)
* 라우팅이 '길을 찾는 과정'이라면, 스위칭은 결정된 경로(인터페이스)로 **패킷을 실제로 통과시켜 전달하는 과정**입니다.
* **주의 :** 라우터도 다음 장비로 패킷을 보낼 때 최종적으로는 2계층 형태의 프레임으로 감싸야 하므로, 상대방의 물리 주소를 알아내기 위해 내부적으로 **ARP(Address Resolution Protocol)**를 수행합니다.

#### 5.2.4 라우팅, 스위칭 우선순위 (경로 선택 기준)
라우팅 테이블에 동일한 목적지로 가는 경로가 여러 개 존재할 경우, 다음의 엄격한 순서대로 우선순위를 판단합니다.

| 순위 | 기준 요소 | 설명 |
| :---: | :--- | :--- |
| **1** | **Longest Match (최장 매치)** | 서브넷 마스크가 **가장 길고 구체적으로 일치**하는 경로를 최우선으로 선택합니다. (ex. `10.1.1.0/24`가 `10.1.0.0/16`보다 우선) |
| **2** | **Administrative Distance (AD, 관리 거리)** | 라우팅 정보의 **신뢰도 값**입니다. 값이 **작을수록** 우선순위가 높습니다. (ex. Connected=0, Static=1, OSPF=110) |
| **3** | **Metric (메트릭)** | **동일한 라우팅 프로토콜 내부**에서 최적 경로를 정하는 기준(비용)입니다. 이 또한 값이 **작을수록** 우선합니다. |

---

### 5.3 라우팅 설정 방법 요약

#### 5.3.1 다이렉트 커넥티드
* 라우터 물리/논리 인터페이스에 IP 주소를 할당하고 활성화(`no shutdown`)하면 라우팅 테이블에 활성화 상태(`C`)로 자동 생성됩니다.

#### 5.3.2 스태틱 라우팅
* 관리자가 경로를 고정하므로 장비의 CPU/RAM 리소스를 아낄 수 있고 보안에 유리하지만, 네트워크 변화에 자동 대응이 불가능합니다.
* **디폴트 루트 (Default Route) :** 라우팅 테이블에 없는 모든 모르는 목적지 패킷을 특정 방향으로 던지기 위해 사용하는 규칙입니다. 스태틱 라우팅 형태로 구현됩니다. (`ip route 0.0.0.0 0.0.0.0 [Next-Hop]`)

#### 5.3.3 다이나믹 라우팅
* **장점 :** 라우터가 토폴로지 변화를 감지해 경로를 자동 학습 및 수정하므로 관리가 편리하고 대규모 환경에 적합합니다.
* **단점 :** 프로토콜 동작 및 라우팅 업데이트 교환을 위해 장비의 **CPU/RAM 리소스 소모가 증가**하며, 초반 프로토콜 설계 및 설정이 상대적으로 복잡합니다.

## 6장 로드 밸런서/방화벽: 4계층 장비(세션 장비)

> **개요:** TCP/UDP 포트(4계층) 정보를 기반으로 동작하며, 단순 패킷 전달을 넘어 '세션(연결 상태)'을 관리하는 장비들을 학습합니다.

### 6.1 4계층 장비의 특징
* **정의:** OSI 7계층 중 4계층(전송 계층)의 **TCP/UDP 헤더**를 확인하고 제어하는 장비입니다.
* **확인 가능한 정보:** 출발지/목적지 IP, 출발지/목적지 Port, TCP 상태(SYN, ACK 등의 제어 플래그)
* **대표 장비:** * **로드 밸런서:** 트래픽 분산 처리
  * **방화벽:** 접근 통제 및 보안

---

### 6.2 로드 밸런서 (Load Balancer)
* **정의:** 서버로 들어오는 요청을 여러 대의 서버에 공평하게 분산하여 처리하는 장비입니다.
* **주요 장점:**
  1. **부하 분산:** 특정 서버의 과부하(Overload) 방지
  2. **장애 대응:** `WEB1`에 장애가 발생하면 `WEB2`, `WEB3`로 트래픽을 우회시켜 서비스 연속성 보장
  3. **확장성:** 서비스 규모 확장 시 서버 추가(Scale-out)가 용이

#### 6.2.1 L4 스위치
* **특징:** TCP/UDP 정보(IP 및 Port)를 기반으로 서버를 선택합니다.
  * *예시:* `443 포트 요청` ➔ `WEB1`, 다음 `443 포트 요청` ➔ `WEB2`
* **장점:** 웹 컨텐츠 내용을 보지 않고 헤더만 보기 때문에 **속도가 빠르고 구조가 단순**합니다.
* **단점:** 애플리케이션 정보(URL, 쿠키, HTTP 헤더 등)를 확인할 수 없습니다.

#### 6.2.2 ADC (Application Delivery Controller)
* **특징:** L7 로드밸런서의 발전된 형태로, 애플리케이션 계층(7계층)까지 확인하여 세밀한 제어가 가능합니다.
* **확인 정보:** URL, 쿠키, HTTP Header, Host 정보 등
  * *예시:* `example.com/login` ➔ `WEB1(로그인 서버)`, `example.com/image` ➔ `IMAGE SERVER`
* **주요 고급 기능:**
  * **SSL 종료 (SSL Termination):** 클라이언트와의 HTTPS 암호화 통신을 ADC가 대신 복호화하여 내부 서버에는 HTTP로 전달 (서버 연산 부하 절감)
  * **캐싱 (Caching):** 정적 파일을 ADC가 직접 응답하여 서버 부하 감소

#### 6.2.3 L4 스위치 vs ADC 비교

| 항목 | L4 스위치 | ADC (L7) |
| :--- | :--- | :--- |
| **동작 계층** | 4계층 | 7계층 |
| **확인 정보** | IP, Port (포트만 확인) | URL, 쿠키, HTTP 헤더 (웹 내용 확인) |
| **처리 속도** | **빠름** | 상대적 느림 (내용물 분석 필요) |
| **URL/쿠키 분석**| ❌ 불가 | ⭕ 가능 |
| **SSL 종료** | 제한적 | ⭕ 가능 |
| **기능 범위** | 단순함 | 매우 다양하고 지능적인 제어 가능 |

---

### 6.3 방화벽 (Firewall)
* **정의:** 허용된 통신만 통과시키고 나머지는 모두 차단하는 **네트워크 필터** 역할을 수행합니다.
* **정책 예시:** * `허용: 10.0.0.0/24 ↔ 443(HTTPS)`
  * `차단: 모든 나머지 통신 (Deny All)`
* **동작 방식의 분류:**
  * **Stateless:** 상태를 기억하지 않고 순수 패킷 정보(IP, Port)만 확인하여 필터링
  * **Stateful (현재 표준):** 패킷뿐만 아니라 통신의 연결 상태인 **세션(Session)**까지 확인하여 보안성 및 처리 효율 증가
* **장점:** 보안 강화, 불필요한 외부 접근 차단

---

### 6.4 4계층 장비를 통과할 때의 유의점

#### 6.4.1 세션 테이블 유지 및 세션 정보 동기화
* **세션(Session)이란?** 장비가 기억하는 TCP 연결 정보입니다.
  * *예시:* `10.0.0.1:50000 -> 8.8.8.8:443` 정보를 로드밸런서나 방화벽의 **세션 테이블**에 기억합니다.
* **필요성:** 올바른 사용자에게 응답 패킷을 정확히 전달하기 위해 필수적입니다.
* **HA(고가용성) 환경에서의 고려사항:**
  * **문제:** 방화벽/로드밸런서를 2대로 이중화했을 때, `장비 1(Active)`에 장애가 발생하여 `장비 2(Standby)`로 인계(`Failover`)되면, `장비 2`에는 기존 세션 정보가 없어 기존 연결들이 끊어집니다.
  * **해결:** 장비 간에 세션 정보를 실시간으로 공유하는 **세션 동기화(Session Synchronization)** 설정이 필요합니다.

#### 6.4.2 비대칭 경로 문제 (실무 빈출!)
* **정상 경로 (대칭):** 요청(`PC ➔ FW1 ➔ 서버`)과 응답(`서버 ➔ FW1 ➔ PC`)이 동일한 방화벽을 통과하므로 정상 통신됩니다.
* **비대칭 경로 (Asymmetric Routing):**
  * 요청: `PC ➔ FW1 ➔ 서버`
  * 응답: `서버 ➔ FW2 ➔ PC`
* **문제점:** 응답 패킷을 받은 `FW2`는 자신의 세션 테이블에 없는 최초 요청(SYN)이 생략된 패킷으로 판단하여 **통신을 강제로 차단**합니다. 결과적으로 통신 실패가 발생합니다.
* **중요성:** 이중화 환경에서 라우팅 경로가 어긋나거나 장애 조치(Failover) 시 자주 발생하므로 정교한 경로 설계가 필요합니다.

#### 6.4.3 하나의 통신에 두 개 이상의 세션이 사용될 때의 고려사항 (★ 매우 중요)
* **대표 예시 (FTP):** FTP 프로토콜은 실제로 하나의 서비스를 위해 **2개의 TCP 연결(세션)**을 사용합니다.
  * **제어 채널 (Control):** 21번 포트 (로그인 및 명령어 전송용)
  * **데이터 채널 (Data):** 별도의 포트 (실제 파일 전송용)
* **문제점:** 방화벽에서 21번 포트만 허용해 두면, 로그인은 정상적으로 되지만 **파일 전송이나 목록 조회 시 데이터 채널이 차단**되어 에러가 발생합니다.
* **해결 방법:** 방화벽에서 **Application Inspection** 또는 **FTP Helper** 기능을 활성화합니다. 이를 통해 방화벽이 제어 채널을 모니터링하다가 데이터 채널이 동적으로 열릴 때 관련 포트를 자동으로 허용해 줍니다.
  
## 7장. 통신을 도와주는 네트워크 주요 기술
실제로 인터넷을 사용하기 위해 필요한 서비스들을 배우는 장이다.

### 7.1 NAT / PAT

#### 7.1.1 NAT/PAT 용도와 필요성
* **NAT (Network Address Translation)**
  * **정의:** IP 주소를 변환하는 기술로, 주로 내부망의 **사설 IP를 공인 IP로 변환**합니다.
  * **필요성:** 전 세계적으로 무한하지 않은 **IPv4 주소 부족 문제를 해결**하기 위해 사용됩니다.
* **PAT (Port Address Translation)**
  * **정의:** NAT의 확장 버전으로, IP 주소뿐만 아니라 **포트(Port) 번호까지 함께 변환**하는 기술입니다.
  * **특징:** 단순 NAT만 사용하면 공인 IP 하나당 PC 한 대만 매핑되어 외부 패킷이 돌아올 때 어느 PC의 것인지 구분하기 어렵습니다. 반면 PAT를 사용하면 포트 번호로 각각의 내부 PC를 식별할 수 있습니다.
  * **실무:** 우리가 가정이나 사무실에서 흔히 쓰는 **인터넷 공유기 대부분은 사실 PAT 방식으로 동작**합니다.

---

#### 7.1.2 NAT 동작 방식
> **기본 원리:** 내부 주소(사설 IP)를 외부 주소(공인 IP)로 변경하여 통신합니다.

1. **요청:** 내부 PC(사설 IP)가 외부 사이트(예: 네이버)에 접속하면 패킷이 NAT 장비(라우터/공인 IP)에 도착합니다.
2. **변환:** NAT 장비는 출발지 사설 IP를 자신의 공인 IP로 변환하여 네이버로 보냅니다.
3. **응답:** 네이버는 사설 IP의 존재를 모른 채 공인 IP가 접속한 것으로 인지하고 응답을 보냅니다. NAT 장비는 이 응답을 다시 원래의 사설 IP로 변환해 내부 PC에 전달합니다.

* **NAT 테이블 예시:**
  | 사설 IP (내부) | 공인 IP (외부) |
  | :--- | :--- |
  | `192.168.0.10` | `203.0.113.100` |

---

#### 7.1.3 PAT 동작 방식
* PAT는 **`IP + 포트 번호`를 세트로 묶어서 변환**합니다.
* 여러 대의 내부 PC가 동시에 외부 사이트에 접속하더라도, 공인 IP는 단 하나만 사용하여 각각 다른 포트 번호를 할당해 트래픽을 정확하게 구분합니다.

---

#### 7.1.4 SNAT와 DNAT
변환하는 대상이 '출발지'인가 '목적지'인가에 따라 분류됩니다.

* **SNAT (Source NAT)**
  * **특징:** 패킷의 **출발지(Source) 주소**를 변경합니다.
  * **용도:** 내부 사설망의 PC들이 인터넷(외부)에 접속할 때 주로 사용됩니다.
* **DNAT (Destination NAT)**
  * **특징:** 패킷의 **목적지(Destination) 주소**를 변경합니다.
  * **용도:** 외부에 있는 사용자가 내부 웹서버 등에 접근할 수 있도록 경로를 열어줄 때 사용되며, 이 구조가 흔히 말하는 **포트 포워딩(Port Forwarding)**입니다.

---

#### 7.1.5 동적 NAT와 정적 NAT
매핑 방식이 고정되어 있는지, 유동적인지에 따라 분류됩니다.

* **정적 NAT (Static NAT)**
  * **방식:** 사설 IP와 공인 IP를 **1:1로 고정 매핑**합니다. (ex. `192.168.0.10` <-> `203.0.113.10` 항상 동일)
  * **특징:** 주소가 고정되어 있으므로 외부에 서비스를 제공하는 **서버 운영에 적합**하며 관리가 직관적입니다.
* **동적 NAT (Dynamic NAT)**
  * **방식:** 장비가 보유한 공인 IP 풀(Pool)에서 **사용 가능한 주소를 임시로 할당**합니다.
  * **특징:** 내부 PC가 인터넷을 요청할 때만 공인 IP를 빌려주고, 접속을 종료하면 다시 풀에 반납합니다. 제한된 공인 IP를 필요할 때만 효율적으로 나누어 쓰므로 주소가 절약됩니다.
   
### 7.2 DNS (Domain Name System)
> DNS는 네트워크와 클라우드를 공부할 때 가장 중요한 서비스 중 하나이며, 실제 대부분의 서비스는 IP가 아닌 DNS를 기반으로 작동합니다.

#### 7.2.1 DNS 소개
* **정의:** `Domain Name System`의 약자로, 사람이 읽기 쉬운 **도메인 이름(문자 주소)을 컴퓨터가 이해하는 IP 주소로 변환**해주는 시스템입니다.

#### 7.2.2 DNS 구조와 명명규칙
> DNS는 전 세계적으로 거대한 **계층형 트리(Tree) 구조**로 관리됩니다.

```text
                . (Root)
                │
       ┌────────┴────────┐
      com               net  (TLD)
       │
     naver                   (Second-Level Domain)
       │
      www                    (Host Name)
```

* **Root Domain:** 최상위 점(`.`)을 의미하며, 실제 인터넷 주소창에서는 보통 생략됩니다.
* **TLD (Top Level Domain):** 최상위 도메인으로 국가나 등록 기관의 목적을 나타냅니다. (ex. `.com`, `.net`, `.org`, `.kr`)
* **Second Level Domain:** 실제 서비스나 회사 이름이 위치하는 단계입니다. (ex. `naver.com`, `google.com`, `amazon.com`)
* **Host Name:** 서비스의 성격(웹, 메일, 블로그 등)을 나타내는 앞부분입니다. (ex. `www.naver.com`, `mail.naver.com`, `blog.naver.com`에서 앞의 호스트명에 해당)
* **FQDN (Fully Qualified Domain Name):** 호스트명부터 최상위 도메인까지 모두 표기한 전체 도메인 이름입니다. (ex. `www.naver.com`)

> 💡 **시험 포인트 (구조 구분)**
> * `www` = 호스트명
> * `naver` = 도메인명
> * `com` = TLD (최상위 도메인)

#### 7.2.3 DNS 동작 방식
1. **사용자 요청:** 사용자가 웹 브라우저 주소창에 `www.naver.com`을 입력합니다.
2. **PC 질의:** PC는 기설정된 로컬 DNS 서버에 해당 도메인의 IP 주소를 알려달라고 요청합니다.
3. **캐시 및 조회:** DNS 서버는 자신의 캐시(Cache)를 먼저 확인하고, 정보가 없다면 최상위인 **Root DNS**부터 조회를 시작합니다.
4. **순차 응답:** `Root DNS` ➔ `com TLD DNS` ➔ `naver 네임서버` 순으로 단계를 밟아가며 최종 IP 주소를 응답받습니다.
5. **IP 획득:** DNS 서버로부터 응답을 받은 사용자 PC가 최종 웹 서버의 IP 주소를 알게 됩니다.
6. **서버 접속:** 알아낸 IP 주소를 바탕으로 TCP 연결을 맺고, HTTP/HTTPS 통신을 통해 네이버 웹 서버에 정상적으로 접속합니다.

#### 7.2.4 마스터와 슬레이브 (Master & Slave)
> DNS 서버가 1대뿐이라면 장애 발생 시 전체 서비스가 중단됩니다. 따라서 안정적인 서비스를 위해 DNS 서버를 여러 대 운영(이중화)해야 합니다.

* **Master DNS:** 원본 데이터를 보유하고 있으며, DNS 설정 파일을 직접 관리합니다.
* **Slave DNS:** Master 서버의 복제본을 가집니다. Master의 데이터를 주기적으로 복사해 와서 동기화합니다.

**동작 방식:**
```text
[ Master DNS ] ─── Zone Transfer (존 전송) ───> [ Slave DNS ]
```
**이중화의 장점:**
1. **가용성 확보:** Master 서버에 장애가 발생해도 Slave 서버가 대신 동작하여 전체 서비스가 중단되지 않고 유지됩니다.
2. **부하 분산:** 전 세계에서 들어오는 수많은 조회(Query) 요청을 여러 서버로 나누어 처리하므로 서버 트래픽 부담이 줄어듭니다.

#### 7.2.5 DNS 주요 레코드 (Record)
> DNS 서버가 도메인에 대해 어떤 정보를 가지고 있는지 정의하는 설정 항목들입니다.

| 레코드 종류 | 역할 | 예시 |
| :--- | :--- | :--- |
| **A** | 도메인 이름을 **IPv4 주소**로 매핑 | `www.naver.com` ➔ `223.130.xxx.xxx` |
| **AAAA** | 도메인 이름을 **IPv6 주소**로 매핑 | `example.com` ➔ `2001:db8::1` |
| **CNAME** | 도메인 이름에 별칭(Alias)을 부여 | `www.company.com` ➔ `server.company.com` |
| **MX** | 해당 도메인의 **메일 서버**를 지정 | `company.com` ➔ `mail.company.com` |
| **NS** | 도메인의 권한이 있는 **DNS 서버(네임서버)를 지정** | `naver.com` ➔ `ns1.naver.com` |
| **PTR** | IP 주소를 도메인 이름으로 변환 (**역방향 조회**) | `223.130.xxx.xxx` ➔ `www.naver.com` |
| **TXT** | 도메인에 대한 텍스트 문자열 저장 | 이메일 인증 및 보안 설정 (`SPF`, `DKIM` 등) |

---

#### 7.2.6 DNS에서 알아두면 좋은 핵심 개념
* **캐시 (Cache):** 한 번 조회한 DNS 결과를 임시 공간에 저장하여, 다음번 동일한 요청이 올 때 네임서버까지 가지 않고 빠르게 응답을 주는 기술입니다.
* **TTL (Time To Live):** DNS 조회 결과가 캐시 내에 유효하게 저장되어 남아있는 시간(초 단위)을 의미합니다.
* **Recursive Query (재귀적 질의):** 클라이언트(PC)가 요청하면 로컬 DNS 서버가 최종 결과(IP 주소)를 다른 서버들로부터 완전히 찾아내어 원스톱으로 제공하는 방식입니다.
* **Iterative Query (반복적 질의):** 최종 답변을 모를 때 "나는 모르니 다음 단계인 이 네임서버로 직접 가보라"며 한 단계씩 다른 네임서버의 힌트(경로)만 제공하는 방식입니다.

---
#### 7.2.7 DNS 설정 (Windows)
* **DNS 정보 및 서버 확인:** `ipconfig /all`
* **DNS 직접 질의:** `nslookup www.naver.com`

---

#### 7.2.8 DNS 설정 (Linux)
* **DNS 서버 확인:** `cat /etc/resolv.conf`
* **DNS 조회 및 상세 확인:** `nslookup google.com` 또는 `dig google.com`

#### 7.2.9 호스트(Hosts) 파일 설정
> DNS 서버를 거치지 않고, **내 로컬 PC 안에서 도메인과 IP를 강제로 직접 매핑**하는 파일입니다. 주로 로컬 개발이나 테스트 환경에서 유용하게 사용됩니다.

* **운영체제별 파일 경로:**
  * **Windows:** `C:\Windows\System32\drivers\etc\hosts`
  * **Linux:** `/etc/hosts`
* **설정 및 접속 예시:**
  ```text
  192.168.0.100   test.local
 ```

### 7.3 GSLB (Global Server Load Balancing)
> **개념 연결하기:** 지금까지 배운 내용을 복습해 보면 **DNS**는 도메인을 IP로 변환하고, **로드밸런서**는 서버로 트래픽을 분산해 주는 역할을 했습니다. 

만약 서비스의 규모가 커져서 서버가 단일 장소가 아닌 **서울 IDC, 부산 IDC, 도쿄 IDC, 싱가포르 IDC 등 전 세계 여러 지역(Global)**에 분산되어 있다면 어떻게 트래픽을 제어해야 할까요? 이때 사용하는 기술이 바로 **GSLB**입니다.

---

#### 7.3.1 GSLB 동작 방식
* **GSLB의 정의:** `Global Server Load Balancing`은 전 세계 여러 데이터센터(또는 서버) 중 **사용자에게 가장 적절하고 최적화된 데이터센터를 선택**하여 연결해 주는 기술입니다.
* **도입 효과:** 서울 사용자는 서울 IDC로, 부산 사용자는 부산 IDC로 다이렉트 접속하게 유도함으로써 **응답 속도 향상, 네트워크 비용 감소, 서비스 장애 대응(고가용성)**이 가능해집니다.

> 💡 **일반 DNS vs GSLB 차이점**
> * **일반 DNS의 문제점:** 고정된 하나의 IP 주소만 응답하기 때문에, 전 세계 모든 사용자가 거리와 상관없이 무조건 동일한 하나의 서버로만 접속하여 병목 현상이 발생합니다.
> * **GSLB 적용 시:** 사용자의 물리적 위치를 파악하여 서울 사용자에겐 서울 서버 IP를, 부산 사용자에겐 부산 서버 IP를 전달하는 등 **사용자 위치 및 환경에 따라 다르게 최적의 IP를 응답**합니다.

---

#### 7.3.2 GSLB 구성 방식

##### 🔹 기본 구조
``` text
      [ 사용자 (Client) ]
               │
               ▼
         [ GSLB (DNS) ]
               │
       ┌───────┴───────┐
       ▼               ▼
  [ 서울 IDC ]    [ 부산 IDC ]

```

##### 🔹 구성 요소
* **사용자 (Client):** 서비스 접속을 요청하는 주체입니다.
* **GSLB 서버:** 사용자의 위치나 서버 상태를 분석하여 트래픽을 어디로 보낼지 결정합니다.
* **데이터센터 (IDC):** 실제 웹 서버나 앱 서버가 위치하여 서비스를 제공하는 인프라 공간입니다.

##### 🔹 GSLB가 확인하는 핵심 정보 (판단 기준)
* **서버 상태 (Health):** 서울 서버가 정상인지, 부산 서버가 장애 상태인지 실시간 체크합니다.
* **응답 속도 (Latency):** 사용자와 각 IDC 간의 네트워크 응답 속도를 측정하여 어느 쪽이 더 빠른지 판단합니다.
* **서버 부하 (Load):** 각 데이터센터의 CPU/메모리 사용률, 세션 수 등을 확인하여 여유가 있는 곳을 고릅니다.
* **사용자 위치 (Geo-Location):** 접속한 사용자의 IP를 기반으로 물리적 위치(서울, 부산, 일본 등)를 파악합니다.

> 💡 **실무 예시**
> 글로벌 대기업이나 클라우드 기반 서비스들은 안정성과 속도를 위해 **서울 리전, 도쿄 리전, 싱가포르 리전 등 전 세계 여러 지역에 서버를 분산 배치**하고 GSLB로 지능형 라우팅을 수행합니다.

---

#### 7.3.3 GSLB 분산 방식
> GSLB가 어떤 기준을 바탕으로 사용자 트래픽을 각 데이터센터로 분배할지 결정하는 알고리즘(방식)입니다.

| 분산 방식 | 작동 원리 및 특징 | 장점 | 단점 / 비고 |
| :--- | :--- | :--- | :--- |
| **1. Round Robin** | 특별한 조건 없이 순서대로 교대하며 IP를 분배하는 가장 단순한 방식 | 구현이 매우 간단함 | 서버의 실시간 상태나 부하를 전혀 고려하지 못함 |
| **2. Geographic** | 사용자의 IP를 기반으로 가장 가까운 지역의 IDC 연결 (ex. 서울 유저는 서울 IDC) | 응답 속도가 매우 빠름 | **실무에서 가장 대중적으로 사용**되는 방식 |
| **3. Health Check** | 서버 상태를 실시간 감시하여 정상 서버로만 연결 (ex. 서울 장애 시 부산으로 자동 전환) | **고가용성(HA)** 확보 | 서비스 중단을 막는 필수 기법 |
| **4. Load Based** | 서버의 자원 상태를 기반으로 분배 (ex. 서울 CPU 부하가 높으면 부산으로 우회) | 서버 자원을 효율적으로 사용 | 실시간 부하를 모니터링하는 오버헤드 발생 |
| **5. Response Time** | 각 IDC와의 응답 속도를 측정해 가장 빠른 곳 연결 (ex. 서울 10ms vs 부산 30ms ➔ 서울 연결) | 사용자 체감 성능 극대화 | 네트워크 환경에 따라 수시로 변동될 수 있음 |

---

#### 7.3.4 DNS와 GSLB의 핵심 차이점 요약

> 일반 DNS는순수한 '주소록' 역할만 수행하는 반면, GSLB는 서버의 상태와 사용자 위치를 모두 고려하는 '지능형 네비게이션' 역할을 합니다.

| 구분 | 일반 DNS | GSLB |
| :--- | :--- | :--- |
| **주요 역할** | 도메인 이름을 단순 IP 주소로 변환 | 사용자에게 가장 최적의 서버/데이터센터 선택 |
| **서버 장애 판단** | ❌ 불가능 (서버가 죽어도 기존 IP 반환) | ⭕ 가능 (장애 발생 시 정상 서버 IP로 우회) |
| **사용자 위치 판단** | ❌ 불가능 (누구에게나 동일한 IP 반환) | ⭕ 가능 (사용자와 가까운 IDC IP 반환) |
| **지능형 부하 분산** | ❌ 불가능 (단순 순서 배정만 가능) | ⭕ 가능 (서버 CPU/트래픽 상태에 따라 분산) |
| **데이터센터 선택** | ❌ 불가능 (단일 IDC 기준 작동) | ⭕ 가능 (전 세계 다중 IDC 간 정교한 라우팅) |

---

### 7.4 DHCP (Dynamic Host Configuration Protocol)

> **도입 배경:** PC나 스마트폰 같은 단말기가 네트워크(인터넷)에 정상적으로 연결되려면 최소한 다음 **4가지 핵심 정보**가 필요합니다.
> 1. **IP 주소 (IP Address)**
> 2. **서브넷 마스크 (Subnet Mask)**
> 3. **기본 게이트웨이 (Default Gateway)**
> 4. **DNS 서버 주소 (DNS Server)**

만약 사내에 관리해야 할 PC가 10,000대라면, 관리자가 10,000번을 돌아다니며 이 정보를 수동으로 직접 입력해야 하므로 매우 비효율적이고 IP가 중복되는 실수가 발생할 수 있습니다. 

이러한 문제를 해결하기 위해 **네트워크 정보를 자동으로 할당하고 관리해 주는 기술인 DHCP**를 사용합니다.

---

#### 7.4.1 DHCP 프로토콜

* **DHCP의 정의:** `Dynamic Host Configuration Protocol`은 네트워크에 접속하려는 장비(PC, 스마트폰 등)에게 IP 주소 및 관련 네트워크 설정을 **자동으로 할당**해 주는 애플리케이션 계층의 프로토콜입니다.
* **DHCP가 할당하는 5대 정보:**
  1. **IP 주소** (IP Address)
  2. **서브넷 마스크** (Subnet Mask)
  3. **기본 게이트웨이** (Default Gateway)
  4. **DNS 서버 주소** (DNS Server)
  5. **임대 시간** (Lease Time): IP를 사용할 수 있는 유효 기간

> 💡 **DHCP 도입 전후 비교**
> * **DHCP가 없을 때:** 사용자가 직접 네트워크 정보를 하나하나 수동으로 입력해야 하며, 오타나 IP 중복 충돌이 발생할 위험이 큽니다.
> * **DHCP가 있을 때:** 랜선을 꽂거나 와이파이에 연결하는 순간 자동으로 설정이 완료되어 즉시 인터넷을 사용할 수 있습니다.

---

#### 7.4.2 DHCP 동작 방식 (DORA 과정)
> DHCP의 핵심은 **DHCP 클라이언트(PC)**와 **DHCP 서버**가 서로 통신하며 IP를 주고받는 것입니다. 이 과정을 각 단계의 앞 글자를 따서 **DORA 과정**이라고 부릅니다.

##### 🔹 DORA 4단계 요약
| 단계 | 명칭 | 통신 방식 | 주요 역할 |
| :--- | :--- | :--- | :--- |
| **1단계** | **D**iscover | 브로드캐스트 | PC가 네트워크에 최초 연결 시, DHCP 서버를 찾기 위해 사방에 외침 ("IP 주소 좀 주세요!") |
| **2단계** | **O**ffer | 유니캐스트/브로드캐스트 | DHCP 서버가 PC에게 임대 가능한 IP 주소를 제안 ("`192.168.10.100` 어때?") |
| **3단계** | **R**equest | 브로드캐스트 | PC가 제안받은 IP를 사용하겠다고 확정 통보 ("방금 제안하신 IP로 쓰겠습니다!") |
| **4단계** | **A**cknowledge | 유니캐스트/브로드캐스트 | DHCP 서버가 최종 승인하며 네트워크 정보 전송 ("확인 완료! 계약 기간 동안 잘 쓰렴~") |

##### 🔹 텍스트 흐름도로 이해하기
```text
  [ DHCP 클라이언트 (PC) ]                      [ DHCP 서버 ]
            │                                         │
            │ ─────── 1. DHCP Discover (방송) ──────> │
            │ <────── 2. DHCP Offer (제안) ────────── │
            │ ─────── 3. DHCP Request (선택) ───────> │
            │ <────── 4. DHCP ACK (최종 승인) ─────── │
            │                                         │
```

#### 7.4.3 DHCP 서버 구성
> DHCP 서버는 네트워크 정보를 단순히 뿌려주는 것이 아니라, 관리자가 설정한 특정 자원 범위 내에서 체계적으로 IP 주소를 제어합니다.

##### 🔹 DHCP 서버 주요 관리 항목 및 예시
| 관리 항목 | 설명 | 설정 예시 |
| :--- | :--- | :--- |
| **IP Pool** | 클라이언트에게 자동으로 할당해 줄 수 있는 주소 범위 | `192.168.10.100 ~ 192.168.10.200` |
| **서브넷 마스크** | 네트워크 구분을 위한 서브넷 마스크 | `255.255.255.0` |
| **기본 게이트웨이** | 외부 네트워크로 나가는 통로 (라우터 IP) | `192.168.10.1` |
| **DNS 서버** | 도메인 해석을 위해 단말이 바라볼 DNS 서버 IP | `8.8.8.8` (주), `1.1.1.1` (보조) |
| **Lease Time** | 발급된 IP 주소를 클라이언트가 사용할 수 있는 유효 시간 | 환경에 따라 수 시간 ~ 수 일 설정 |

##### 🔹 주요 특징 및 기능
* **IP 충돌 방지:** DHCP 서버가 내부적으로 어떤 IP가 지급되었는지 실시간으로 기록하고 관리하므로, 동일한 IP가 두 번 할당되어 발생하는 **IP 중복 충돌을 완벽하게 방지**합니다.
* **예약 할당 (Reservation):** 자동 할당이 원칙이지만, 특정 단말의 고유 식별 주소인 **MAC 주소를 고정 매핑**하여 항상 일정한 IP만 발급받도록 설정할 수 있습니다.
  * *실무 활용:* 네트워크 주소가 수시로 바뀌면 곤란한 **사내 프린터, CCTV, NAS, 서버 장비** 등은 예약 할당 기능을 활용해 고정 IP처럼 사용합니다.

---

#### 7.4.4 DHCP 릴레이 (DHCP Relay Agent)

* **🚨 문제 상황 (네트워크의 한계):** DHCP의 첫 단계인 `DHCP Discover`는 **브로드캐스트(Broadcast)** 통신을 사용합니다. 하지만 **라우터(Router)는 브로드캐스트 패킷이 다른 네트워크 영역으로 넘어가지 못하도록 차단**하는 성질을 가지고 있습니다. 
  따라서 서로 다른 서브넷(네트워크 대역)에 PC와 DHCP 서버가 분리되어 있다면, PC의 요청이 라우터에 막혀 DHCP 서버에 도달하지 못하는 문제가 발생합니다.
  > `PC` ➔ `[브로드캐스트 차단]` ➔ `라우터` ❌ `DHCP 서버`

* **💡 해결 방법:** 이 문제를 해결하기 위해 라우터에 **DHCP 릴레이(DHCP Relay Agent)** 기능을 활성화합니다.

##### 🔹 동작 메커니즘
1. **PC (Discover 전송):** PC가 브로드캐스트로 IP 할당 요청을 보냅니다.
2. **라우터 수신:** 라우터가 이 브로드캐스트를 수신합니다.
3. **유니캐스트 변환:** 라우터(DHCP 릴레이)가 해당 브로드캐스트 패킷을 목적지가 명확한 **유니캐스트(Unicast) 패킷으로 변환**하여 멀리 있는 DHCP 서버로 원격 전달합니다.
4. **서버 응답:** DHCP 서버가 정상적으로 응답을 돌려줍니다.
5. **IP 할당 완료:** 라우터를 거쳐 PC가 최종적으로 IP를 할당받습니다.

##### 🔹 텍스트 흐름도로 이해하기
```text
       [ PC (Client) ]
              │
              │ 📢 Broadcast (Discover)
              ▼
    [ Router (DHCP Relay) ]
              │
              │ 🎯 Unicast (원격 전달)
              ▼
       [ DHCP Server ]

```

## 8장. 서버 네트워크 기본

### 8.1 서버의 네트워크 설정 및 확인

서버가 네트워크를 통해 통신하기 위해서는 반드시 아래 **4가지 핵심 정보**가 설정되어 있어야 합니다.

| 항목 | 역할 | 예시 |
| :--- | :--- | :--- |
| **IP 주소 (IP Address)** | 네트워크 상에서 서버 **자신을 식별하기 위한 고유 주소** | `192.168.10.100` |
| **서브넷 마스크 (Subnet Mask)** | IP 주소에서 **동일한 네트워크 영역**이 어디까지인지 구분하는 기준 | `255.255.255.0` |
| **게이트웨이 (Gateway)** | 내부 네트워크를 벗어나 **외부 인터넷으로 나가는 출구 (라우터 IP)** | `192.168.10.1` |
| **DNS 서버 (DNS Server)** | `google.com` 같은 도메인 이름을 **컴퓨터가 이해하는 IP 주소로 변환** | `8.8.8.8` |

---

### 🌐 외부 네트워크 통신 흐름 (예시: google.com 접속 시)

서버가 외부 인터넷 영역에 있는 `google.com`에 접속할 때는 내부적으로 다음과 같은 단계를 거칩니다.

```text
[서버] ──(1. DNS 질의)──> [DNS 서버] (google.com의 IP 주소 확인)
  │
  ├──(2. IP 확인) ──> 확인된 목적지 IP가 나와 "같은 네트워크"인가?
  │                      ├── [YES] ──> 내부 통신 진행
  │                      └── [NO] ──> (3. 게이트웨이로 패킷 전송)
  ▼
[게이트웨이(라우터)] ──(4. 외부망 라우팅)──> [인터넷] ──> [google.com]
```
1. **DNS 질의**: `google.com`이라는 이름의 IP 주소를 DNS 서버에 요청합니다.
2. **IP 확인**: DNS 서버로부터 도메인에 매핑된 실제 IP 주소를 받아와 확인합니다.
3. **같은 네트워크인지 확인**: 서버의 서브넷 마스크를 이용해 목적지 IP가 나와 동일한 로컬 네트워크 대역에 있는지 비교합니다.
4. **게이트웨이 전송**: 확인 결과 목적지가 외부 네트워크라면, 외부망 접점인 기본 게이트웨이(Gateway)로 패킷을 전송합니다.
5. **인터넷 통신**: 게이트웨이를 거쳐 라우팅된 패킷이 인터넷 망을 통과해 최종 목적지와 통신을 수행합니다.


### 8.1.1 리눅스 서버 네트워크

#### 1. 네트워크 확인 명령어
* **현재 IP 확인**: `ip addr` 또는 `ip a`
* **게이트웨이 확인**: `ip route`
* **DNS 확인**: `cat /etc/resolv.conf`
* **네트워크 인터페이스 확인**: `ip link`
* **포트 확인**: `ss -tulnp`

#### 2. 네트워크 연결 및 상태 점검
* **네트워크 연결 확인 (ping)**: `ping 8.8.8.8` (상대방과 외부 통신이 가능한지 확인)
* **DNS 작동 확인**: `ping google.com` (도메인으로 ping이 성공하면 DNS 정상 작동)
* **lo (Loopback) 인터페이스**: `127.0.0.1` (외부망을 거치지 않고 서버 자기 자신과 통신할 때 사용)

---

#### 🛠️ 실무 예시: 웹 서버 접속이 안 될 때 점검 순서
1. **IP 확인** (`ip a`)
2. **게이트웨이 확인** (`ip route`)
3. **DNS 확인** (`cat /etc/resolv.conf`)
4. **Ping 테스트** (`ping 8.8.8.8` 및 `ping google.com`)
5. **포트 상태 확인** (`ss -tulnp`)
6. **방화벽 설정 확인**

### 8.1.2 윈도 서버 네트워크

#### 1. 네트워크 확인 명령어
* **필요한 기본 정보**: IP, 서브넷 마스크, 게이트웨이, DNS
* **IP 주소 및 상세 정보 확인**: `ipconfig` 또는 `ipconfig /all`
* **네트워크 연결 확인 (ping)**: `ping 8.8.8.8`
* **DNS 질의 및 작동 확인**: `nslookup google.com`
* **라우팅 테이블(게이트웨이) 확인**: `route print`
* **포트 상태 및 PID 확인**: `netstat -ano`

## 8.2 서버의 라우팅 테이블

서버도 라우팅 테이블을 가지고 있다.

### 8.2.1 서버의 라우팅 테이블

* **라우팅 테이블이란?** 서버가 패킷을 어디로 보낼지 결정하기 위해 사용하는 경로 정보입니다.
* 쉽게 말하면 **목적지가 어디냐? 어느 인터페이스로 보낼까? 어느 게이트웨이로 보낼까?**를 기록한 표입니다.

#### 🔄 서버가 판단하는 과정
1. **목적지 확인**
2. **라우팅 테이블 검색**
3. **일치하는 경로 선택**
4. **인터페이스 결정**
5. **패킷 전송**

#### 📋 대표적인 라우팅 테이블 예시

| Destination | Gateway | 비고 |
| :--- | :--- | :--- |
| `192.168.10.0/24` | `Direct` | 해당 대역은 게이트웨이를 거치지 않고 **직접 전송** |
| `0.0.0.0/0` | `192.168.10.1` | 그 외 모든 목적지는 **기본 경로(Default Route)**로 전송 |


### 8.2.2 리눅스 서버의 라우팅 확인 및 관리

#### 1. 라우팅 테이블 확인
* **명령어**: `ip route` 또는 `route -n`
* **출력 예시**:
  ```text
  default via 192.168.10.1 dev ens33
  192.168.10.0/24 dev ens33 proto kernel
  ```
  
* **의미**:
  * `default via 192.168.10.1` ──> 기본 게이트웨이
  * `192.168.10.0/24` ──> 직접 연결된 네트워크

#### 2. 특정 목적지 경로 확인
* **명령어**: `ip route get 8.8.8.8`
* **결과**: `8.8.8.8 via 192.168.10.1 dev ens33`
* **의미**: `8.8.8.8`은 게이트웨이로 전송됨

#### 3. 정적 라우트(Static Route) 관리
* **정적 라우트 추가**: `sudo ip route add 172.16.10.0/24 via 192.168.10.254`
  * *의미: `172.16.10.0/24`로 가려면 `192.168.10.254`로 보내라*
* **정적 라우트 삭제**: `sudo ip route del 172.16.10.0/24`

> 💡 **실무 예시**
> 서버가 본사망, 지사망, 백업망 3개 네트워크에 연결될 수 있다. 이때 어디로 보내야 하는지를 라우팅 테이블이 결정한다.

---

### 8.2.3 윈도 서버의 라우팅 확인 및 관리

#### 1. 라우팅 테이블 확인
* **명령어**: `route print`

#### 2. 정적 라우트(Static Route) 관리
* **정적 라우트 추가**: `route add 172.16.10.0 mask 255.255.255.0 192.168.10.254`
  * *의미: `172.16.10.0/24` 대역으로 갈 때 `192.168.10.254`를 경유하라*
* **영구 저장 (-p 옵션)**: `route -p add 172.16.10.0 mask 255.255.255.0 192.168.10.254`
  * *주의: 옵션이 없으면 서버 재부팅 시 설정이 초기화됩니다.*

> ⚠️ **실무 장애 유형: 비대칭 라우팅 (Asymmetric Routing)**
> 실무에서 자주 발생하는 "Ping 안 됨" 증상은 라우팅 문제인 경우가 많습니다.
> * **원인**: 출발지 A에서 목적지 B로 가는 경로는 존재하지만, 돌아오는 목적지 B에서 출발지 A로 오는 경로가 없거나 다르게 잡혀있을 때 발생합니다. 이를 **비대칭 라우팅**이라고 합니다.

## 8.3 네트워크 확인을 위한 명령어

> 🛠️ **실무에서 장애가 발생했을 때 가장 먼저 사용하는 핵심 명령어들입니다.**

### 8.3.1 ping
* **역할**: ICMP 프로토콜을 사용하여 상대 장비와 통신이 가능한지 확인합니다.
* **명령어**:
  * **Linux / Windows 공통**: `ping 8.8.8.8`
* **확인 가능 항목**: 상대 장비 활성화 여부, 네트워크 연결 상태, 지연 시간(Latency)
* **💡 실무 예시**: `ping google.com`은 실패하지만 `ping 8.8.8.8`은 성공한다면 **DNS 설정 문제**를 의심할 수 있습니다.

### 8.3.2 tcping (Windows)
* **역할**: `ping`(ICMP)이 방화벽으로 인해 차단되어 있을 때, TCP 포트가 열려 있는지 확인합니다.
* **명령어 예시**: `tcping 192.168.10.10 80` (80번 포트로 접속이 가능한가?)
* **💡 실무 활용**: 웹 서버 상태 확인 (`tcping www.example.com 443`으로 HTTPS 포트 점검)

### 8.3.3 traceroute (Linux) / tracert (Windows)
* **역할**: 패킷이 목적지까지 도달하는 전체 경로(라우팅 경로)를 추적합니다.
* **명령어**:
  * **Linux**: `traceroute 8.8.8.8`
  * **Windows**: `tracert 8.8.8.8`
* **💡 실무 활용**: 네트워크 연결이 **어느 구간(구간 라우터)에서 끊기는지** 확인하여 장애 지점을 파악합니다.

### 8.3.4 tcptraceroute
* **역할**: `traceroute` 인터페이스의 TCP 기반 버전입니다.
* **사용 이유**: 일반 `traceroute`는 ICMP나 UDP를 사용하므로 중간 방화벽에 차단될 확률이 높습니다. 방화벽 우회를 위해 TCP 포트를 기반으로 경로를 추적합니다.
* **명령어 예시**: `tcptraceroute google.com 443`

### 8.3.5 netstat (Network Statistics)
* **역할**: 시스템의 네트워크 연결 상태 및 열려 있는 포트를 확인합니다.
* **명령어**:
  * **Linux**: `netstat -tunlp`
  * **Windows**: `netstat -ano`
* **💡 실무 활용**: 현재 서버가 특정 서비스 포트를 정상적으로 활성화(Listen)하고 있는지 점검합니다.

### 8.3.6 ss (Socket Statistics)
* **역할**: Linux 환경에서 기존 `netstat`를 대체하는 더 빠르고 효율적인 명령어입니다. (최신 Linux 기본 권장)
* **명령어**: `ss -tulnp`
* **옵션 의미**:

| 옵션 | 의미 |
| :--- | :--- |
| **t** | TCP 프로토콜 확인 |
| **u** | UDP 프로토콜 확인 |
| **l** | LISTEN (대기) 상태의 포트만 표시 |
| **n** | 서비스 이름 대신 숫자로 포트 표시 |
| **p** | 해당 포트를 사용하는 프로세스(PID) 표시 |

### 8.3.7 nslookup (Name Server Lookup)
* **역할**: DNS 서버에 질의하여 도메인 네임의 IP 주소를 조회합니다.
* **명령어**: `nslookup google.com`
* **💡 실무 예시**: 웹 페이지 접속 실패 시, DNS 서버가 올바른 IP 주소를 반환하는지 검증합니다.

### 8.3.8 telnet (Tele Network)
* **역할**: 원격지 서버의 특정 TCP 포트가 열려 있고 접근 가능한지 직관적으로 테스트합니다.
* **명령어 예시**: `telnet 192.168.10.10 80`
* **💡 실무 활용**: 웹 서버(80/443), 데이터베이스 포트(3306/1521), 메일 서버(SMTP/25) 등의 접근성 원격 점검

### 8.3.9 ipconfig (Windows)
* **역할**: Windows 호스트의 네트워크 설정 정보를 확인합니다.
* **명령어**: `ipconfig` (기본 확인) / `ipconfig /all` (상세 정보 확인)
* **확인 가능 항목**: IP 주소, 서브넷 마스크, 게이트웨이, DNS 서버, MAC 주소
* **💡 실무 활용**: 사용자 PC나 Windows 기반 서버에서 네트워크 문제 발생 시 가장 먼저 실행하는 기본 도구입니다.

### 8.3.10 tcpdump
* **역할**: 명령줄(CLI) 기반의 네트워크 패킷 캡처 도구로, Linux의 Wireshark 역할을 합니다.
* **명령어 활용**:
  * **전체 패킷 캡처**: `sudo tcpdump`
  * **특정 인터페이스 지정**: `sudo tcpdump -i eth0`
  * **특정 포트 지정**: `sudo tcpdump port 80`
  * **특정 IP 지정**: `sudo tcpdump host 192.168.10.10`
  * **파일로 저장**: `sudo tcpdump -w capture.pcap`

### 🚨 실무 장애 대응: 웹 서비스 접속 불가 시 표준 확인 순서

실무에서 웹 애플리케이션 접속 장애가 발생하면 아래 순서대로 단계를 좁혀가며 원인을 찾습니다.

1. **`ping`** ──> 대상 서버와 기본적인 네트워크 연결(ICMP)이 살아있는가?
2. **`traceroute` / `tracert`** ──> 통신 패킷이 중간 통신망 어느 구간에서 막히는가?
3. **`nslookup`** ──> 도메인 주소가 올바른 IP로 매핑되어 리턴되는가? (DNS 확인)
4. **`telnet`** ──> 목적지 웹 서버의 포트(80/443)까지 TCP 연결 시도가 닿는가?
5. **`netstat` / `ss`** ──> (서버 내부 진단) 웹 서버 프로세스가 포트를 정상적으로 Listen 중인가?
6. **`tcpdump`** ──> 패킷 레벨에서 요청이 들어오고 나가는지 최종 트래픽 분석 진행

## 9장 보안

### 9.1 보안의 개념과 정의

#### 9.1.1 정보 보안의 정의
* **개념:** 정보를 안전하게 보호하는 모든 활동 (유출 및 변조 방지)
* **보호 대상:** 고객 개인정보, 회사 기밀문서, 서버 데이터, 데이터베이스(DB) 정보 등

#### 💡 정보 보안의 3대 요소 (CIA triad)
| 요소 | 의미 | 실무 예시 |
| :--- | :--- | :--- |
| **C**onfidentiality (기밀성) | 허가된 사람만 정보에 접근 가능 | 로그인, 접근 권한 설정, 데이터 암호화 |
| **I**ntegrity (무결성) | 데이터가 임의로 변경되거나 변조되지 않음 | 파일 전송 중 변조 방지 (해시 체크 등) |
| **A**vailability (가용성) | 필요할 때 언제든 서비스를 사용할 수 있음 | 은행 서비스의 24시간 중단 없는 운영 |

---

###  9.1.2 네트워크의 정보 보안

#### 네트워크 보안이란?
> 네트워크를 통해 이동하는(Transit) 데이터를 외부의 위협으로부터 보호하는 것

* **주요 보호 대상:**
  * **데이터:** 주민등록번호, 계좌번호, 개인정보 등
  * **서버:** 웹 서버, DB 서버, 메일 서버
  * **네트워크 장비:** 스위치, 라우터, 방화벽
  * **사용자 계정:** ID, 비밀번호
* **🚨 보안 부재 시 위험:** 데이터 이동 경로 중간에서 패킷 탈취(Sniffing) 및 변조 가능
* **🛡️ 해결 기술:** `HTTPS`, `VPN`, `IPSec`, `SSL/TLS` 등 암호화 통신 프로토콜 적용

---

### 9.1.3 네트워크 보안의 주요 5대 개념

1. **인증 (Authentication)**
   * **정의:** "당신은 누구인가?"를 확인하는 절차
   * **예시:** ID/비밀번호, OTP, 지문 인식
2. **인가 (Authorization)**
   * **정의:** 신원이 확인된 사용자에게 "어떤 권한을 줄 것인가?"를 결정
   * **예시:** 일반 직원은 '조회만 가능', 관리자는 '수정/삭제까지 가능'
3. **암호화 (Encryption)**
   * **정의:** 데이터를 제3자가 알아볼 수 없게 변환하여 스니핑을 방지
   * **예시:** `SSL/TLS`, `HTTPS`, `IPSec`
4. **접근 제어 (Access Control)**
   * **정의:** 정책에 따라 허용된 사용자/IP/포트만 네트워크에 접근하도록 통제
   * **예시:** 방화벽(Firewall) 정책 설정
5. **감사 (Audit)**
   * **정의:** "누가, 언제, 어떤 행동을 했는지" 추적할 수 있도록 행위를 기록
   * **예시:** 시스템 로그(Log) 수집 및 분석

---

### 9.1.4 네트워크 정보 보안의 발전 추세

#### 🔄 패러다임의 변화
* **과거 (경계 기반 보안):** 회사 내부망은 안전하다고 가정 ➡️ 회사 경계에 **방화벽**만 설치하면 끝
* **현재 (분산 환경 보안):** 클라우드, 재택근무, 모바일 모빌리티, VPN, SaaS 환경 도입으로 '안전한 내부'라는 개념이 사라짐

#### 🚀 최신 보안 트렌드 키워드
* **Zero Trust (제로 트러스트)**
  * *"아무도 믿지 않는다. 항상 검증한다."*
  * 내부 사용자나 장비도 신뢰하지 않고, 접속할 때마다 신원과 권한을 매번 철저히 검증하는 방식
* **MFA (Multi-Factor Authentication, 다중 인증)**
  * 아이디/비밀번호(지식) 외에 OTP(소유), 휴대폰/지문 인증(생체) 등 **2가지 이상의 서로 다른 요소**를 조합하여 인증 강도를 높이는 기술

### 9.2 보안 솔루션의 종류

#### 9.2.1 DDoS 방어 장비
* **DDoS 정의:** `Distributed Denial of Service` (분산 서비스 거부 공격)
* **공격 방식:** 수많은 좀비 PC를 동원해 대량의 트래픽을 유발, 서버 과부하로 서비스를 마비시킴
* **장비 역할:** 비정상적인 트래픽을 실시간으로 감지하고 차단하여 정상 서비스를 보호

#### 9.2.2 방화벽 (Firewall)
* **역할:** 미리 정의된 보안 규칙에 따라 허용할 트래픽과 차단할 트래픽을 결정하는 장비
* **동작 방식:** 네트워크를 통과하는 패킷의 IP/Port 등 정보를 검사하고 정책과 비교 후 허용(Allow) 또는 차단(Drop)

#### 9.2.3 IDS vs IPS (네트워크 침입 방어)
> **💡 한 줄 비유:** IDS는 감시하는 **CCTV**, IPS는 범인을 물리적으로 막는 **경비원**

* **IDS (Intrusion Detection System, 침입 탐지 시스템)**
  * **역할:** 네트워크 트래픽을 모니터링하여 악의적인 공격을 **탐지(경고)**함 (실시간 차단은 불가)
* **IPS (Intrusion Prevention System, 침입 방지 시스템)**
  * **역할:** 공격을 **탐지함과 동시에 실시간으로 차단**까지 수행

#### 9.2.4 WAF (Web Application Firewall, 웹 방화벽)
* **역할:** 일반 방화벽이 보지 못하는 **HTTP/HTTPS 웹 요청 데이터의 내용(L7 영역)을 깊숙이 검사**
* **주요 방어:** SQL Injection, Cross-Site Scripting(XSS) 등 웹 취약점 공격 방어

#### 9.2.5 샌드박스 (Sandbox)
* **역할:** 외부에서 유입된 악성 의심 파일(또는 악성코드)을 가상의 격리된 안전한 공간에서 실제로 실행해 보고 행동을 분석하는 기술
* **특징:** 실제 내부 시스템에 영향을 주지 않고 안전하게 악성 여부 판단 가능

#### 9.2.6 NAC (Network Access Control, 네트워크 접근 제어)
* **역할:** 내부 네트워크에 접속하려는 장비(PC, 모바일 등)가 안전한지 검증하고 제어
* **기능:** 사내 PC가 백신을 켰는지, OS 업데이트를 했는지 체크하여 무결성이 검증된 단말만 네트워크 접속 허용

#### 9.2.7 IP 제어 & 접근 통제
* **IP 기반 접근 통제:** 특정 소스(Source) IP만 접근할 수 있도록 제한하는 방식 (예: AWS의 보안 그룹(Security Group) 정책)
* **접근 통제 (Access Control):** "누가, 어디에, 어떤 권한으로" 접근 가능한지 정책적으로 결정하는 보안 메커니즘

#### 9.2.8 VPN (Virtual Private Network, 가상 사설망)
* **개념:** 공용 인터넷망을 마치 우리 회사만 쓰는 안전한 **전용 회선(Private Line)**처럼 사용할 수 있게 해주는 기술
* **핵심 특징 3가지:**
  1. **터널링 (Tunneling):** 인터넷 구간에 안전한 가상 통로를 생성
  2. **암호화 (Encryption):** 터널을 통과하는 데이터를 암호화하여 스니핑 방지
  3. **인증 (Authentication):** 인가된 정당한 사용자만 VPN에 접속할 수 있도록 확인

### 9.3 방화벽 (Firewall)

#### 9.3.1 방화벽의 정의
* **개념:** 네트워크 출입구(경계면)에서 들어오고 나가는 모든 트래픽을 검사하여, 보안 정책에 따라 통과시키거나 차단하는 가장 기본적인 네트워크 보안 장비

#### 9.3.2 초기 방화벽 vs 현대적 방화벽
* **초기 방화벽 (Packet Filtering, 패킷 필터링 방식)**
  * **검사 항목:** 출발지/목적지 IP, 포트 번호, 프로토콜
  * **한계:** 패킷을 하나하나 개별적으로만 검사하기 때문에, 앞뒤 패킷 간의 연관성이나 정상적인 연결 흐름(Context)을 파악하지 못하는 문제점이 있음
* **현대적 방화벽 (SPI, Stateful Packet Inspection, 상태 기반 패킷 검사)**
  * **특징:** 단순히 패킷 자체만 보는 것이 아니라, TCP Handshake 과정 등 **연결 상태(Session) 정보까지 기억**하고 추적하는 엔진 탑재
  * **장점:** 정상적인 세션 연결을 통해 들어오는 데이터는 정책 검사를 매번 새로 하지 않고 빠르게 통과시킴

#### 9.3.3 방화벽의 일반적인 동작 방식
1. **패킷 수신:** 네트워크 경계에서 패킷이 방화벽으로 들어옴
2. **세션/SPI 엔진 확인:** 이미 연결되어 있는 기존 세션(Session Table)에 해당하는 패킷인지 먼저 확인
3. **보안 정책 확인:** 새로운 연결 요청이라면, 등록된 방화벽 허용/차단 규칙(Rule)을 위에서부터 순서대로 대조
4. **통과 또는 차단:** 정책에 맞으면 통과(Allow/Permit), 맞지 않으면 차단(Drop/Deny)

#### 9.3.4 ALG (Application Layer Gateway)
* **개념:** FTP, SIP 같이 데이터 전송을 위해 동적으로 포트를 열어 써야 하는 **특수 프로토콜을 방화벽이 제대로 인식하고 처리할 수 있도록 돕는 기능**
* **역할:** 방화벽 뒤에 숨은 사설 IP를 공인 IP로 변환할 때(NAT) 애플리케이션 페이로드 내부의 IP 정보까지 찾아서 함께 변경해 줌

#### 9.3.5 🚨 방화벽의 한계점
> 방화벽은 '문지기' 역할일 뿐, 허가된 문으로 들어오는 교묘한 공격까지 완벽하게 막지는 못합니다.

1. **허용된 트래픽의 취약성:** 웹 서비스를 위해 `80(HTTP)`이나 `443(HTTPS)` 포트를 열어두면, 해커가 이 포트를 타고 들어오는 공격 트래픽도 무조건 통과됨
2. **내부자/내부망 공격:** 외부 경계선에만 방화벽이 있기 때문에, 내부 직원의 PC가 악성코드에 감염되어 내부 서버를 공격하는 경우 방화벽을 거치지 않아 막을 수 없음
3. **암호화 트래픽 검사 불가:** `HTTPS`, `VPN`, `SSL/TLS` 등으로 암호화된 트래픽은 데이터 알맹이가 암호화되어 있어 방화벽이 패킷 내부를 들여다보고 검사하기 어려움
4. **애플리케이션 계층 공격 방어 취약:** L3/L4 중심의 일반 방화벽은 `SQL Injection`, `XSS(Cross-Site Scripting)`, 악성 파일 업로드 등 L7(웹 애플리케이션) 영역의 정교한 공격을 막기 어려움 *(➡️ 이를 보완하기 위해 **WAF(웹 방화벽)**나 **IPS**를 병행 사용)*


## 9.4 IPS와 IDS

### 9.4.1 IPS / IDS의 정의 및 차이점
* **IDS (Intrusion Detection System, 침입 탐지 시스템)**
  * **역할:** 네트워크에 대한 공격이나 비정상적인 흐름을 **탐지하고 관리자에게 알림(Alert)**하는 시스템
  * **구성 방식 (미러링 포트):** 실제 통신 경로 밖에서 패킷 복사본을 받아 분석하는 **오프라인(Passive) 방식**
    * *경로:* `인터넷` ➡️ `방화벽` ➡️ `L3/L4 스위치` ➡️ `서버` (스위치에서 패킷을 복사하여 `IDS`로 전달)
  * **장단점:** IDS 장비에 장애가 발생해도 실제 서비스 트래픽에는 영향을 주지 않지만, 공격을 실시간으로 직접 차단할 수는 없음

* **IPS (Intrusion Prevention System, 침입 방지 시스템)**
  * **역할:** 공격을 **탐지함과 동시에 실시간으로 차단(Block)**까지 수행하는 능동형 시스템
  * **구성 방식 (In-Line):** 실제 패킷이 오고 가는 통신 경로 상에 직접 배치되는 **인라인(Active) 방식**
    * *경로:* `인터넷` ➡️ `방화벽` ➡️ `IPS` ➡️ `스위치` ➡️ `서버` (모든 패킷이 IPS를 통과함)
  * **장단점:** 탐지 즉시 차단이 가능해 보안성이 높지만, IPS 장비 자체에 오동작이나 과부하(장애)가 생기면 전체 네트워크가 마비될 위험이 있음

---

### 9.4.2 IPS / IDS 동작 방식 및 차단 메커니즘

#### 1. 탐지 방식 (Detection Method)
* **시그니처 기반 탐지 (Signature-based):** 기존에 알려진 공격 패턴(규칙) 데이터베이스를 보유하고 있다가, 유입되는 패킷 데이터와 일치하는 패턴이 있는지 검사하는 방식 (알려진 공격 방어에 효과적)
* **이상 탐지 (Anomaly-based):** 평소 정상적인 트래픽의 행동 패턴을 학습(Baseline 수립)한 뒤, 이를 벗어나는 비정상적인 급증이나 행위가 발생하면 공격으로 간주하는 방식 (알려지지 않은 신종 공격 방어에 효과적)

#### 2. IPS의 주요 차단 방식 (Prevention Actions)
1. **Packet Drop (패킷 버림):** 악성으로 판단된 해당 패킷을 목적지로 전달하지 않고 중간에서 즉시 버림
2. **TCP Reset (세션 강제 종료):** 공격자와 피해자 양쪽에 `RST(Reset)` 패킷을 보냄으로써 맺어져 있던 TCP 세션을 강제로 끊어버림
3. **IP 차단 (Blacklisting):** 공격을 유발한 출발지 IP 주소를 블랙리스트에 등록하여, 일정 시간 동안 해당 IP로부터 오는 모든 접속을 원천 차단
4. **Rate Limit (트래픽 속도 제한):** 디도스(DDoS) 공격처럼 비정상적으로 많은 트래픽이 몰릴 때, 대역폭이나 초당 패킷 수를 제한하여 시스템 다운을 막음

---

### 9.4.3 기존 시스템의 한계와 극복 (NGIPS)

> **전통적인 IPS/IDS의 한계**
> 현대의 트래픽은 대부분 암호화(`HTTPS`)되어 있어 패킷 내부를 들여다보기 어렵고, 하루가 다르게 변하는 신종 취약점(Zero-day)과 교묘한 고도화 공격을 단순 패턴 매칭만으로 막기에는 성능적·기능적 한계가 존재합니다.

이를 해결하기 위해 등장한 것이 바로 **NGIPS (Next-Generation IPS, 차세대 침입 방지 시스템)**입니다.

* **애플리케이션(서비스) 인지:** 단순히 포트 번호(예: 80, 443)만 보는 것이 아니라, 실제로 그 포트에서 실행되는 구체적인 서비스와 애플리케이션 행위를 정밀 분석
* **사용자 기준 정책 제어:** 고정된 IP 주소가 아닌, 사내 dynamic IP 환경에서도 '사용자 ID/계정' 기준으로 누가 어떤 데이터에 접근하는지 분석하고 정책 적용
* **실시간 위협 인텔리전스:** 전 세계의 최신 공격 정보 및 취약점 데이터를 클라우드를 통해 자동으로 실시간 업데이트 받아 방어
* **머신러닝 기반 분석:** 인공지능/머신러닝을 활용하여 기존 시그니처에 없는 정교하고 지능적인 이상 행위와 변종 공격을 탐지
* **지능형 악성코드 탐지 (Sandboxing 연동):** 단순 네트워크 패킷 분석을 넘어, 유입되는 파일의 악성 여부까지 추적하고 탐지하는 고도화된 보안 기능 제공

## 9.5 DDoS 방어 장비

DDoS 방어 장비는 대량의 악성 트래픽으로부터 네트워크 인프라와 주요 서비스를 보호하는 핵심 보안 솔루션입니다.

### 9.5.1 DDoS 방어 장비의 정의
* **DDoS (Distributed Denial of Service, 분산 서비스 거부 공격):** 감염된 여러 대의 공격 시스템(봇넷)이 동시에 특정 서버나 네트워크에 처리 능력을 초과하는 과도한 트래픽을 보내 서비스를 마비시키는 공격입니다.
* **DDoS 방어 장비의 역할:** 인라인 또는 아웃오브패스(Out-of-Path) 방식으로 트래픽을 감시하여, 이러한 비정상 트래픽만 정밀하게 식별·차단하고 **정상 사용자의 요청은 중단 없이 처리되도록 보장**합니다.

---

### 9.5.2 DDoS 방어 장비 동작 방식
방어 장비는 네트워크로 유입되는 모든 트래픽을 실시간으로 모니터링하며, 사전에 정해진 정상 패턴(Baseline)과 비교 분석하여 공격을 완화합니다.

1. **트래픽 수집:** 네트워크 게이트웨이 단계에서 유입되는 패킷 및 세션 데이터를 실시간으로 수집
2. **이상 징후 분석:** 트래픽의 급격한 임계치 초과, 프로토콜 비정상 행위 등 정상 범위를 벗어난 흐름 감지
3. **공격 식별:** 유입된 트래픽이 어떤 유형의 DDoS 공격(볼류메트릭, 프로토콜, 애플리케이션 등)인지 명확히 분류
4. **차단 및 완화:** 공격 트래픽은 드롭(Drop) 또는 차단하고, 정상 트래픽만 필터링하여 내부 서버로 안전하게 전달

---

### 9.5.3 DDoS 공격 유형 (Attack Types)
DDoS 공격은 공격자가 목표로 삼는 타깃 자원을 고갈시키는 방식에 따라 크게 세 가지 유형으로 분류됩니다.

| 공격 유형 | 주요 타깃 자원 | 공격 특징 |
| :--- | :--- | :--- |
| **볼류메트릭 공격**<br>(Volumetric) | 네트워크 대역폭 (Bandwidth) | 회선 용량을 초과하는 대량의 패킷을 쏟아부어 네트워크 자체를 포화시킴 |
| **프로토콜 공격**<br>(Protocol) | 네트워크 장비 및 서버 인프라 자원 | TCP/IP 프로토콜의 구조적 취약점이나 연결 방식(Handshake)을 악용해 세션 테이블이나 장비 처리 능력을 고갈시킴 |
| **애플리케이션 공격**<br>(Application) | 웹/애플리케이션 서버 내부 자원 | 정상적인 HTTP 요청처럼 위장하여 과도한 요청을 보냄으로써 웹 서버의 CPU, 메모리, 스레드 풀 등을 소모시킴 |

---

### 9.5.4 볼류메트릭 공격 (Volumetric Attack)
볼류메트릭 공격은 네트워크 파이프라인(대역폭) 자체를 막아버리는 가장 직관적이고 위협적인 형태의 대규모 DDoS 공격입니다.

#### 1. 주요 특징
* **대역폭 소진:** 서비스 가입 회선 용량(예: 10Gbps)을 초과하는 트래픽을 발생시켜, 정상적인 사용자의 패킷이 아예 통로에 진입하지 못하게 만듭니다.
* **대량 분산 트래픽:** 전 세계에 분산된 악성코드 감염 장비(Zombie PC, IoT 기기 등)로 구성된 대규모 봇넷을 이용해 동시다발적으로 공격을 수행합니다.
* **대표적인 예시:** UDP Flood, ICMP Flood, 그리고 취약한 오픈 서버를 악용해 응답 크기를 수십~수백 배로 키우는 **증폭 반사 공격(Reflection/Amplification Attack)** 등이 있습니다.

#### 2. 실무에서의 방어 및 완화 방법
* **트래픽 속도 제한 (Rate Limiting):** 특정 대역, 국가, 또는 단일 IP 및 프로토콜별로 유입되는 초당 요청량(PPS/BPS)의 최대 임계치를 설정하여 제한합니다.
* **필터링 정책 적용:** 무의미한 UDP/ICMP 패킷이나 악성 패턴이 확인된 시그니처 기반의 패킷을 네트워크 진입로에서 조기에 드롭(Drop)합니다.
* **DDoS 스크러빙 센터 (Scrubbing Center) 활용:** 대규모 공격이 발생했을 때 트래픽을 클라우드 기반의 대용량 정화 센터로 우회(BGP 백홀링 등)시켜, 공격 트래픽은 걸러내고 클린 트래픽만 원래 서버로 안전하게 바이패스(Bypass)시킵니다.
* **다계층 방어 체계 (Multi-layered Defense):** 전용 DDoS 방어 장비뿐만 아니라 방화벽(FW), 침입방지시스템(IPS), 웹 방화벽(WAF)을 유기적으로 연동하여 하위 네트워크 계층부터 상위 애플리케이션 계층까지 통합적으로 보호합니다.

## 9.6 VPN (Virtual Private Network)

VPN은 공용 인터넷 망을 마치 기업 전용선처럼 안전하게 사용할 수 있도록 **가상의 안전한 터널을 구축하는 기술**입니다.

### 9.6.1 VPN의 종류
VPN은 연결 목적과 대상에 따라 크게 두 가지 형태로 분류됩니다.

#### 1. Site-to-Site VPN (지점 간 VPN)
* **정의:** 본사와 지사, 사설 데이터센터(IDC) 간, 또는 온프레미스 인프라와 AWS/Azure 같은 클라우드 환경을 상호 연결하는 방식입니다.
* **특징:** **네트워크와 네트워크를 연결**하는 구조이며, 주로 장비 대 장비(Gateway to Gateway) 형태로 상시 연결되어 있습니다.

#### 2. Remote Access VPN (원격 접속 VPN)
* **정의:** 재택근무자나 출장 중인 임직원이 외부에서 회사 내부 네트워크에 안전하게 접속할 수 있도록 지원하는 방식입니다.
* **특징:** **개인 사용자(단말)와 회사 네트워크를 연결**하는 구조이며, 필요할 때만 세션을 연결합니다.

> **💡 구현 프로토콜에 따른 분류**
> * **SSL VPN:** 별도의 복잡한 프로그램 설치 없이, 일반적인 **웹 브라우저 기반**으로 편리하게 보안 접속을 제공합니다. (주로 Remote Access에 활용)
> * **IPSec VPN:** 네트워크 계층(IP Layer)에서 패킷 자체를 강력하게 **암호화하여 전송**하는 방식입니다. (Site-to-Site 및 고보안 Remote Access에 활용)

---

### 9.6.2 VPN의 핵심 동작 원리
VPN은 공공 인터넷망에서 데이터의 안전성을 보장하기 위해 크게 **3가지 핵심 메커니즘**을 수행합니다.

1. **터널링 (Tunneling)**
   * 원래의 패킷 외부를 새로운 헤더로 감싸는 **캡슐화(Encapsulation)** 기술입니다. 
   * 공용 인터넷 환경에서 출발지와 목적지 사이에 전용 통로(터널)가 개설된 것처럼 패킷을 안전하게 이동시킵니다.
2. **암호화 (Encryption)**
   * 터널을 통과하는 데이터(패킷)를 제3자가 훔쳐보더라도 내용을 알 수 없도록 복잡한 알고리즘을 통해 암호화합니다. 이를 통해 데이터의 기밀성을 보장합니다.
3. **인증 (Authentication)**
   * 데이터가 전송되는 과정에서 위·변조되지 않았는지 검증(무결성)하고, 허가된 정당한 사용자나 장비만 VPN 터널에 접근할 수 있도록 신원을 확인합니다.
     
## 10장 서버의 방화벽 설정 및 동작

### 10.1 리눅스 서버의 방화벽 확인 및 관리
리눅스 운영체제는 외부 인프라 장비(하드웨어 방화벽)와 별개로, 서버 자체적으로 네트워크 트래픽을 제어할 수 있는 호스트 기반 방화벽 기능을 제공합니다. 대표적인 도구로는 `iptables`, `firewalld`, `nftables` 등이 있습니다.

#### 10.1.1 iptables 이해하기
* **개념:** 리눅스 커널 내부의 패킷 필터링 프레임워크인 **Netfilter**를 제어하는 명령어입니다. 쉽게 말해 서버가 수신·송신하는 패킷을 어떤 기준으로 허용하거나 차단할지 규칙(Rule)을 정의하는 프로그램입니다.

* **핵심 동작 구조 (기본 Chain):** 패킷이 서버에 도달하면 목적에 따라 아래의 체인을 순차적으로 통과하며 검사받습니다.
  * **INPUT:** 외부에서 **서버 본체로 들어오는** 트래픽 검사 (가장 많이 설정됨)
  * **OUTPUT:** 서버 본체에서 **외부로 나가는** 트래픽 검사
  * **FORWARD:** 서버를 거쳐 **다른 목적지로 전달(라우팅)**되는 트래픽 검사 (서버가 라우터나 게이트웨이 역할을 할 때 사용)

* **패킷 처리 정책 (Target Action):** 매칭되는 규칙을 발견했을 때 수행할 동작입니다.
  * **ACCEPT:** 패킷의 통과를 **허용**합니다.
  * **DROP:** 패킷을 **조용히 버립니다.** (송신자에게 아무런 응답을 주지 않아 타임아웃을 유발하므로 보안상 유리)
  * **REJECT:** 패킷을 차단하고, 송신자에게 **거부 응답 패킷(ICMP 에러 등)을 명시적으로 반환**합니다.

---

#### 10.1.2 리눅스 방화벽 활성화 / 비활성화
CentOS/RHEL 계열에서 주로 사용하는 `firewalld` 데몬을 제어하는 기본 관리 명령어입니다.

* **현재 방화벽 상태 확인:** `systemctl status firewalld`
* **방화벽 서비스 시작(활성화):** `systemctl start firewalld`
* **부팅 시 자동 실행 설정:** `systemctl enable firewalld`
* **방화벽 서비스 중지(비활성화):** `systemctl stop firewalld`

---

#### 10.1.3 리눅스 방화벽 정책 확인
현재 커널에 적용되어 있는 iptables 필터링 규칙 목록을 조회하는 방법입니다.

* **기본 규칙 목록 확인:** `iptables -L`
* **상세 보기 (실무 권장):** IP 주소를 도메인이 아닌 숫자로 빠르게 표시하고(-n), 매칭된 패킷 수와 인터페이스 정보(-v)까지 함께 출력합니다. ➡️ `iptables -L -n -v`

---

#### 10.1.4 리눅스 방화벽 정책 관리 (규칙 추가)
`iptables` 명령어를 통해 특정 포트와 IP에 대한 접근 제어 규칙을 추가(-A INPUT)하는 실무 예제입니다.

* **SSH(22번 포트) 허용:** 외부에서 서버로 들어오는 TCP 22번 포트 접근 승인
  ➡️ `iptables -A INPUT -p tcp --dport 22 -j ACCEPT`
* **HTTP(80번 포트) 허용:** 웹 서비스 제공을 위한 TCP 80번 포트 접근 승인
  ➡️ `iptables -A INPUT -p tcp --dport 80 -j ACCEPT`
* **특정 악성/공격 IP 차단:** 192.168.1.200 번호로부터 오는 모든 패킷을 응답 없이 차단
  ➡️ `iptables -A INPUT -s 192.168.1.200 -j DROP`

---

#### 10.1.5 리눅스 방화벽 로그 확인
침입 시도 파악이나 방화벽 오작동 트러블슈팅을 위해 차단/허용 로그를 모니터링하는 것은 매우 중요합니다.

* **시스템 로그 데몬을 통한 로그 실시간 확인:** `journalctl -xe`
* **전통적인 커널/시스템 메시지 로그 파일 실시간 추적:** `tail -f /var/log/messages`


## 10.2 윈도 서버의 방화벽 확인 및 관리
리눅스에서 `iptables`를 사용했다면, 윈도우 환경에서는 **Windows Defender Firewall(정식 명칭: 고급 보안이 설정된 Windows 숙성 방화벽)**을 사용합니다. 외부 하드웨어 방화벽과 별개로 서버 자체에서 인입/유출되는 패킷을 제어하는 호스트 기반 방화벽입니다.

---

### 10.2.1 윈도 방화벽 활성화/비활성화
* **방화벽의 개념:** 윈도우 서버의 네트워크 인터페이스 앞단에서 패킷을 검사하여 정책에 따라 허용(Allow) 또는 차단(Drop/Block)을 결정하는 핵심 보안 기능입니다.
* **GUI(그래픽 인터페이스) 실행 방법:** * `윈도우 키 + R` 입력 ➡️ `wf.msc` 입력 후 엔터 (고급 보안이 설정된 Windows Defender 방화벽 창이 열립니다.)
* **PowerShell(CLI)을 통한 활성화/비활성화 (실무 팁):**
  * **모든 프로필 방화벽 켜기:** `Set-NetFirewallProfile -All -Enabled True`
  * **모든 프로필 방화벽 끄기:** `Set-NetFirewallProfile -All -Enabled False`

---

### 10.2.2 윈도 방화벽 정책 확인
윈도우 방화벽은 트래픽의 방향에 따라 크게 두 가지 규칙으로 분리하여 관리합니다.

* **인바운드(Inbound) 규칙:** 외부에서 **윈도우 서버 본체로 들어오는** 트래픽에 대한 제어 규칙 (예: 웹 서버의 80/443 포트 개방, RDP 3389 포트 접근 허용 등)
* **아웃바운드(Outbound) 규칙:** 윈도우 서버 본체에서 **외부로 나가는** 트래픽에 대한 제어 규칙 (기본값은 모두 허용인 경우가 많으나, 보안 강화를 위해 특정 목적지 출력을 제한할 때 설정)

---

### 10.2.3 윈도 방화벽 정책 관리
GUI 환경(`wf.msc`)의 우측 '작업' 창을 통해 규칙을 추가/삭제하거나, PowerShell 명령어로 신속하게 정책을 제어할 수 있습니다.

* **PowerShell을 이용한 인바운드 규칙 추가 예시:**
  * **웹 서비스(TCP 80) 허용 규칙 추가:**
    ```powershell
    New-NetFirewallRule -Name "Allow_HTTP" -DisplayName "HTTP 웹 서비스 허용" -Direction Inbound -Protocol TCP -LocalPort 80 -Action Allow
    ```
  * **원격 데스크톱(RDP 3389) 특정 IP만 허용 (보안 강화):**
    ```powershell
    New-NetFirewallRule -Name "Allow_RDP_Specific" -DisplayName "특정 IP RDP 허용" -Direction Inbound -Protocol TCP -LocalPort 3389 -RemoteAddress 192.168.1.50 -Action Allow
    ```

---

### 10.2.4 윈도 방화벽 로그 확인
침입 시도 분석 및 네트워크 트러블슈팅을 위해 차단/허용된 패킷의 로그를 확인하는 방법입니다.

* **로그 활성화 및 경로 설정:**
  1. `wf.msc` 실행 ➡️ 중앙의 **[Windows Defender 방화벽 속성]** 클릭
  2. 도메인/개인/공용 프로필별로 [로그] 항목의 **[사용자 지정]** 버튼 클릭
  3. '낙하된 패킷 로그(Log dropped packets)' 및 '성공한 연결 로그'를 **[예(Yes)]**로 변경
* **기본 로그 파일 저장 경로:**
  * ➡️ `%systemroot%\system32\LogFiles\Firewall\pfirewall.log` (일반적으로 `C:\Windows\System32\LogFiles\Firewall\pfirewall.log`)
